## §1. Vấn Đề Với Callbacks — ES5 Approach

### 1.1 Bài Viết: "No Serious Way of Tracking"

> Will: _"It's a bit ODD that you could have TONS of background work going on, and within JavaScript, no serious way of tracking the work you're doing. That feels... not ideal."_

```
ES5 APPROACH — CÁC VẤN ĐỀ:
═══════════════════════════════════════════════════════════════

  setTimeout(printHello, 1000);
  // → Sends work to browser...
  // → Returns... basically NOTHING useful in JS!
  // → (just a timer ID)
  // → Browser does ALL the work!
  // → JS has NO CLUE what's happening!

  VẤN ĐỀ ①: KHÔNG tracking trong JS!
  ┌──────────────────────────────────────────────────┐
  │ setTimeout(fn1, 1000);                           │
  │ setTimeout(fn2, 2000);                           │
  │ xhrRequest(url, fn3);                            │
  │                                                  │
  │ → 3 background tasks đang chạy!                 │
  │ → JS: "Tôi KHÔNG BIẾT chúng đang ở đâu!"      │
  │ → Không có object nào track progress!            │
  │ → Không biết khi nào xong!                      │
  │ → Không biết có lỗi không!                      │
  │ → = BLIND DELEGATION! 🙈                        │
  └──────────────────────────────────────────────────┘

  VẤN ĐỀ ②: Data chỉ available trong callback!
  ┌──────────────────────────────────────────────────┐
  │ xhrRequest("tiktok.com/videos", function(data) { │
  │   // data CHỈ CÓ ở đây!                        │
  │   // Muốn dùng data ở ngoài? KHÔNG ĐƯỢC!       │
  │   displayVideos(data);                           │
  │                                                  │
  │   // Muốn data cho API call khác?                │
  │   xhrRequest(data.nextUrl, function(moreData) {  │
  │     // data + moreData CHỈ ở đây!              │
  │     processMore(moreData);                       │
  │                                                  │
  │     xhrRequest(moreData.url, function(evenMore) { │
  │       // → NESTING DEEPER AND DEEPER! 💀        │
  │     });                                          │
  │   });                                            │
  │ });                                              │
  └──────────────────────────────────────────────────┘

  Will: "Our response data — the only place it's going
  to be available is in the function that runs on
  completion. That was a bit of a PROBLEM."

## §3. ES6 Solution: Promises — "Two-Pronged Facade Functions"

### 3.1 Bài Viết: "Two Prongs"

> Will: _"In ES6, we got introduced promises. These are what I call TWO-PRONGED facade functions. They have consequences BOTH within the web browser AND they also return a placeholder in JavaScript — a special type of object known as a promise."_

```
TWO-PRONGED FACADE FUNCTION:
═══════════════════════════════════════════════════════════════

  ES5 — ONE PRONG (setTimeout style):
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  setTimeout(fn, 1000)                            │
  │       │                                          │
  │       └──→ ① BROWSER: start timer!              │
  │                                                  │
  │  → Chỉ CÓ 1 prong: work goes to browser!       │
  │  → NOTHING meaningful in JS!                     │
  │  → JS is BLIND! 🙈                              │
  │                                                  │
  └──────────────────────────────────────────────────┘

  ES6 — TWO PRONGS (fetch style):
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  const futureData = fetch("tiktok.com/videos")   │
  │       │                           │              │
  │       │                           └──→ ② BROWSER:│
  │       │                                 network! │
  │       │                                 speak to │
  │       │                                 TikTok!  │
  │       │                                          │
  │       └──→ ① JAVASCRIPT:                        │
  │            Return Promise object!                │
  │            futureData = { value: undefined,       │
  │                           onFulfilled: [] }      │
  │            = PLACEHOLDER for future data!        │
  │                                                  │
  │  → 2 PRONGS: browser work + JS placeholder!     │
  │  → JS has a REFERENCE to track the work!         │
  │  → = NO MORE blind delegation!                  │
  │                                                  │
  └──────────────────────────────────────────────────┘

  Will: "One prong goes off to the web browser,
  the other prong returns an immediate object
  into futureData."
```

> Will: _"I think FETCH does the MOST STUFF of any five letters in JavaScript. It's going to kick off background work speaking to TikTok. At the SAME TIME it's going to have an effect directly in JavaScript."_

```
fetch — DUAL EFFECT:
═══════════════════════════════════════════════════════════════

  const futureData = fetch("https://tiktok.com/api/videos");

  PRONG ① — JAVASCRIPT (immediate!):
  ┌──────────────────────────────────────────────────┐
  │ → Returns a PROMISE object immediately!          │
  │ → futureData = {                                 │

  PRONG ② — WEB BROWSER (background!):
  ┌──────────────────────────────────────────────────┐
  │ → Kicks off NETWORK REQUEST!                     │
  │ → Speaking to TikTok server!                     │
  │ → May take 100ms, 300ms, who knows!             │
  │ → When response comes back:                      │
  │   → Updates futureData.value with response!      │
  │   → Runs functions in onFulfilled array!         │
  └──────────────────────────────────────────────────┘

  TIMELINE:
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │ TIME  JS                      BROWSER            │
  │ ────  ──                      ───────            │
  │ 0ms   futureData = Promise    Network: → TikTok  │
  │       { value: undefined }    Sending request... │
  │                                                  │
  │ 1ms   (continue other code!)  Still waiting...   │
  │ ...                           Still waiting...   │
  │                                                  │
  │ 300ms (maybe idle)            Response! 📦       │
  │       futureData.value =      → Send data to JS  │
  │         { videos: [...] }                        │
  │       Run onFulfilled fns!                       │
  │                                                  │
  └──────────────────────────────────────────────────┘

## §5. Promise Object — Placeholder Cho Data Tương Lai

```
PROMISE OBJECT — CẤU TRÚC:
═══════════════════════════════════════════════════════════════

  const futureData = fetch("url");

  futureData (Promise object):
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  {                                               │
  │    value: undefined          ← data goes HERE!   │
  │                                 (khi response    │
  │                                  comes back)     │
  │                                                  │
  │    onFulfilled: []           ← functions to run  │
  │                                 WHEN data arrives│
  │                                                  │
  │    status: "pending"         ← current state     │
  │                                 "pending" →      │
  │                                 "fulfilled" or   │
  │                                 "rejected"       │
  │  }                                               │
  │                                                  │
  └──────────────────────────────────────────────────┘

  2 BENEFITS:

  ① TRACKING within JS!
  ┌──────────────────────────────────────────────────┐
  │ → We have an OBJECT in JS that represents work!  │
  │ → Can check: futureData.status → "pending"?     │
  │ → Can attach handlers: futureData.then(fn)!     │
  │ → KNOW that background work is happening!        │
  │ → No more blind delegation!                     │
  │                                                  │
  │ Will: "Know that we've set up some background    │
  │ work and KEEP TRACK of the fact that we've       │
  │ done that within JavaScript. That's quite nice!" │
  └──────────────────────────────────────────────────┘

  ② READABLE code!
  ┌──────────────────────────────────────────────────┐
  │ // ES5 (callback hell):                          │
  │ xhrGet(url, function(data) {                     │
  │   xhrGet(data.next, function(more) {             │
  │     display(more);                               │
  │   });                                            │
  │ });                                              │
  │                                                  │
  │ // ES6 (promises — linear!):                     │
  │ fetch(url)                                       │
  │   .then(data => fetch(data.next))                │
  │   .then(more => display(more));                  │
  │                                                  │
  │ → LINEAR! Đọc từ trên xuống!                    │
  │ → No nesting!                                    │
  │ → "Looks more linear and intuitive"             │
  └──────────────────────────────────────────────────┘
```

## §6. Vẫn Async Under The Hood — "Very Similar Pattern!"

### 6.1 Bài Viết: Will's Warning

> Will: _"Even though this code LOOKS more linear and intuitive, under the hood, we're going to have a very, very SIMILAR pattern to what we've just seen. This is still a law of JavaScript — it's synchronous HERE. Background work comes back in asynchronously."_

```
UNDER THE HOOD — SAME MODEL!:
═══════════════════════════════════════════════════════════════

  SURFACE (looks linear):
  ┌──────────────────────────────────────────────────┐
  │ const futureData = fetch("url");                 │
  │ futureData.then(display);                        │
  │ console.log("Me first!");                        │
  │                                                  │
  │ → Looks like: fetch → display → log              │
  │ → Intuitive! Linear! Nice!                      │
  └──────────────────────────────────────────────────┘

  REALITY (same async model!):
  ┌──────────────────────────────────────────────────┐
  │ 0ms: fetch → Promise object + browser request!   │
  │ 0ms: .then → store display in onFulfilled!       │
  │ 1ms: console.log("Me first!") → runs FIRST!     │
  │ ... waiting for network...                       │
  │ 300ms: Data arrives! → display(data) runs!       │
  │                                                  │
  │ OUTPUT:                                          │
  │ 1. "Me first!"     ← sync code first!           │
  │ 2. (display runs)   ← async, after data arrives! │
  │                                                  │
  │ → SAME PATTERN as setTimeout!                    │
  │ → Sync first, async later!                       │
  │ → Just LOOKS nicer on the surface!               │
  └──────────────────────────────────────────────────┘

  Will: "I would argue promises actually OBSCURE even
  more what's really going on, because under the hood,
  it's going to ALL be just the same as this — well,
  with some key new queues."

  KEY NEW QUEUES:
  → Promises KHÔNG dùng callback queue!
  → Promises dùng MICROTASK queue!
  → Microtask queue = HIGHER PRIORITY!
  → SẼ HỌC chi tiết ở bài tiếp theo!
```

```javascript
// CHỨNG MINH 2: Build a simple Promise from scratch!

function MyPromise() {
  this.value = undefined;
  this.status = "pending";
  this.onFulfilled = [];
  this.onRejected = [];
}

MyPromise.prototype.then = function (onFulfill, onReject) {
  if (this.status === "fulfilled") {
    // Already resolved! Run immediately:
    onFulfill(this.value);
  } else {
    // Not yet! Store for later:
    this.onFulfilled.push(onFulfill);
    if (onReject) this.onRejected.push(onReject);
  }
  return this; // allow chaining!
};

MyPromise.prototype.resolve = function (value) {
  this.value = value;
  this.status = "fulfilled";
  // Run ALL stored handlers:
  this.onFulfilled.forEach((fn) => fn(value));
};

// Usage — TWO-PRONGED!
function myFetch(url) {
  // PRONG 1 — JS: return Promise immediately!
  const promise = new MyPromise();

  // PRONG 2 — "Browser": kick off background work!
  setTimeout(() => {
    // Simulate network response:
    const data = { url, videos: ["cat.mp4", "dog.mp4"] };
    console.log("[Browser] Network response arrived!");
    promise.resolve(data); // update value + run handlers!
  }, 300);

  return promise; // returned IMMEDIATELY!
}

// Use it:
console.log("Start");
const futureData = myFetch("tiktok.com/api");
// futureData = { value: undefined, status: "pending" }!
console.log("futureData.value:", futureData.value); // undefined!

futureData.then((data) => {
  console.log("Display:", data.videos);
});

console.log("Me first!"); // runs BEFORE data arrives!

// Output:
// "Start"
// "futureData.value: undefined"    ← placeholder!
// "Me first!"                      ← sync first!
// ... 300ms later ...
// "[Browser] Network response arrived!"
// "Display: ['cat.mp4', 'dog.mp4']" ← async later!
```

```javascript
// CHỨNG MINH 3: Callbacks vs Promises — side by side

// CALLBACK HELL:
function getDataCallback(url, cb) {
  setTimeout(() => cb({ user: "Will", id: 105 }), 100);
}

getDataCallback("api/user", function (user) {
  getDataCallback("api/orders/" + user.id, function (orders) {

  .then((orders) => getDataPromise("api/details/" + orders))
  .then((details) => console.log("Promise result:", details));
// → LINEAR! No nesting! Readable!
```

## §8. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 8.1 Pattern ①: 5 Whys — Promises

```
5 WHYS: TẠI SAO CẦN PROMISES?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao callbacks không đủ?
  └→ Vì KHÔNG tracking trong JS!
     Background work = invisible!
     Data chỉ available trong callback!

  WHY ②: Tại sao cần placeholder object?
  └→ Vì developer cần BIẾT: "work is happening!"
     Promise = đại diện cho work trong JS!
     Có trạng thái: pending → fulfilled/rejected!

  WHY ③: Tại sao "two-pronged"?
  └→ Vì CẦN cả hai: browser work + JS tracking!
     Prong 1: browser does network/timer/etc.
     Prong 2: JS gets a Promise to work with!

  WHY ④: Tại sao .then() thay vì nested callbacks?
  └→ Vì .then() = LINEAR chaining!
     Không nesting!
     Đọc từ trên xuống, trái qua phải!
     = More readable, maintainable!

  WHY ⑤: Tại sao vẫn async under the hood?
  └→ Vì JS vẫn SINGLE-THREADED!
     Browser vẫn làm work bên ngoài!
     Event loop vẫn controls execution order!
     Promises = nicer API, SAME mechanism!
```

### 8.2 Pattern ②: First Principles

```
FIRST PRINCIPLES — PROMISES/A+ SPEC:
═══════════════════════════════════════════════════════════════

  25.6 Promise Objects (ECMAScript Spec):

  → "A Promise is an object used as a placeholder
     for the eventual results of a deferred
     (and possibly asynchronous) computation."

  3 STATES:
  → pending: initial state, not fulfilled or rejected
  → fulfilled: operation completed successfully
  → rejected: operation failed

  TRANSITIONS:
  → pending → fulfilled (ONCE, irreversible!)
  → pending → rejected (ONCE, irreversible!)
  → fulfilled/rejected = SETTLED (never changes again!)

  KEY DIFFERENCES from callbacks:
  → Promise = VALUE (can be stored, returned, passed!)
  → Callback = CODE (can only be called)
  → Promise = OBJECT trong JS (trackable!)
  → Callback = FUNCTION passed away (invisible!)
```

### 8.3 Pattern ③: Trade-off Analysis

```
TRADE-OFFS: CALLBACKS vs PROMISES:
═══════════════════════════════════════════════════════════════

  ┌──────────────────┬──────────────────┬──────────────────┐
  │                  │ CALLBACKS (ES5)  │ PROMISES (ES6)   │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Tracking         │ ❌ None!         │ ✅ Promise object│
  │ Readability      │ ❌ Nesting hell  │ ✅ Linear chain  │
  │ Error handling   │ ❌ Each callback │ ✅ .catch() chain│
  │ Composability    │ ❌ Hard          │ ✅ Promise.all   │
  │ Understanding    │ ✅ Explicit!     │ ❌ "Magic" 🪄    │
  │ Performance      │ ✅ Slightly less │ ⚠️ Object alloc  │
  │ Simplicity       │ ✅ Direct        │ ❌ More concepts │
  │ Queue used       │ Callback queue   │ Microtask queue! │
  │ Priority         │ Lower            │ HIGHER! ⬆️       │
  └──────────────────┴──────────────────┴──────────────────┘

  Will: "I would argue promises actually OBSCURE
  even more what's really going on — because under
  the hood, it's going to all be just the same."
```

### 8.4 Pattern ④: Mental Mapping

```
MENTAL MAP: CALLBACKS → PROMISES → ASYNC/AWAIT:
═══════════════════════════════════════════════════════════════

  EVOLUTION:

  ES5 — CALLBACKS:
  ┌──────────────────────────────────────────────────┐
  │ xhrGet(url, function(data) {                     │
  │   xhrGet(url2, function(data2) {                 │
  │     display(data2);                              │
  │   });                                            │
  │ });                                              │
  │ → Explicit but UGLY!                             │
  └──────────────────────────────────────────────────┘
           ↓ evolution

  ES6 — PROMISES:
  ┌──────────────────────────────────────────────────┐
  │ fetch(url)                                       │
  │   .then(data => fetch(url2))                     │
  │   .then(data2 => display(data2));                │
  │ → Linear but still ASYNC!                       │
  └──────────────────────────────────────────────────┘
           ↓ evolution

  ES2017 — ASYNC/AWAIT:
  ┌──────────────────────────────────────────────────┐
  │ async function getData() {                       │
  │   const data = await fetch(url);                 │
  │   const data2 = await fetch(url2);               │
  │   display(data2);                                │
  │ }                                                │
  │ → LOOKS synchronous but IS async!               │
  │ → Sugar for Promises!                            │
  └──────────────────────────────────────────────────┘

  ALL THREE = SAME under the hood:
  → Browser does work
  → Event loop manages callbacks
  → Just SYNTAX improvements!
```

### 8.5 Pattern ⑤: Reverse Engineering

```javascript
// TỰ BUILD: Two-pronged facade function concept

class TwoProngedFetch {
  constructor() {
    this.browserWork = []; // track browser side
  }

  // Two-pronged facade function:
  fetch(url) {
    console.log(`\n=== fetch("${url}") ===`);

    // PRONG 1 — JavaScript: return Promise immediately!
    const promise = {
      value: undefined,
      status: "pending",
      onFulfilled: [],
      then(fn) {
        if (this.status === "fulfilled") {
          fn(this.value);
        } else {
          this.onFulfilled.push(fn);
        }
        return this; // chaining!
      },
    };
    console.log(`  [PRONG 1 — JS] Promise created!`);
    console.log(`    value: ${promise.value}`);
    console.log(`    status: "${promise.status}"`);

    // PRONG 2 — Browser: kick off network request!
    console.log(`  [PRONG 2 — Browser] Network request started!`);
    const work = {
      url,
      startTime: Date.now(),
      promise, // reference to update later!
    };
    this.browserWork.push(work);

    // Simulate network response (browser thread!):
    setTimeout(() => {
      const responseData = {
        url,
        videos: ["trending.mp4", "funny.mp4"],
        timestamp: Date.now(),
      };
      console.log(`\n  [Browser] Response from ${url}!`);
      promise.value = responseData;
      promise.status = "fulfilled";
      promise.onFulfilled.forEach((fn) => fn(responseData));
    }, 300);

    // Return IMMEDIATELY (prong 1)!
    return promise;
  }
}

// Usage:

  console.log(`Display: ${data.videos.join(", ")}`);
});

console.log("Me first! (sync code runs before data!)");
```
