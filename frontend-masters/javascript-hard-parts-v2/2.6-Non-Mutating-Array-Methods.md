## §1. Vấn Đề: reverse, sort, splice — 3 "Tội Phạm" Mutating

### 1.1 Bài Viết: "Tội Ác" Từ Ngày Đầu

Will Sentance gọi đây là "crime" — 3 array methods có từ JavaScript 1.0 (1995) mà **THAY ĐỔI MẢNG GỐC** thay vì tạo mảng mới:

> _"Some of the built-in major methods in JavaScript actually do the work on the array directly. Key ones: reverse, sort, and splice."_

```javascript
// 3 "TỘI PHẠM" — MUTATING METHODS:

const array1 = [1, 2, 3];

// ❌ reverse() — ĐẢO mảng GỐC!
array1.reverse();
console.log(array1); // [3, 2, 1] ← BỊ ĐỔI!

// ❌ splice() — CẮT/CHÈN vào mảng GỐC!
array1.splice(1, 1, 6); // tại vị trí 1, xoá 1, chèn 6
console.log(array1); // [3, 6, 1] ← BỊ ĐỔI!

// ❌ sort() — SẮP XẾP mảng GỐC!
array1.sort();
console.log(array1); // [1, 3, 6] ← BỊ ĐỔI!

// Mảng gốc [1, 2, 3] đã HOÀN TOÀN BIẾN MẤT! 😱
```


## §2. Tại Sao Mutation = Xấu? (Especially UI)

### 2.1 Bài Viết: Đừng Sửa Data Gốc — Đặc Biệt Trong UI

Will giải thích rất sâu sắc:

> _"Most of the time, we do NOT want to reverse our original underlying data. We're going to use that data to be displayed or passed to some other evaluation, not to actually change the intrinsic data."_

> _"Especially in a language that's a lot of UI — things users see. A lot of the time it's not trying to manipulate the data itself, but to take the data and manipulate it for DISPLAY."_

```
TẠI SAO MUTATION = XẤU? ĐẶC BIỆT TRONG UI:
═══════════════════════════════════════════════════════════════

  VÍ DỤ THỰC TẾ: Spotify Playlist
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  const songs = ["Bohemian Rhapsody", "Stairway to Heaven", "Hotel California"];

  ❌ User click "Sort Z→A" → songs.reverse()
  → Data GỐC bị đảo!
  → User click "Sort A→Z" lại → songs.reverse() lần nữa
  → Nếu có BUG ở giữa → data bị hỏng vĩnh viễn!
  → Nếu nhiều components DÙNG CHUNG songs → tất cả bị ảnh hưởng!

  ✅ User click "Sort Z→A" → displaySongs = songs.toReversed()
  → Data GỐC songs = NGUYÊN VẸN!
  → displaySongs = mảng MỚI để hiển thị
  → User click "Sort A→Z" → lại lấy từ songs gốc!
  → An toàn, predictable, không side effects!

  ┌─────────────────────────────────────────────────────────┐
  │ Will: "You don't want to change underlying data,        │
  │ because you're constantly changing how the user sees it, │
  │ but please don't change the underlying."                 │
  │                                                         │
  │ RULE: Data gốc = SOURCE OF TRUTH                        │
  │       Hiển thị = DERIVED (tạo mới từ gốc)              │
  │       → KHÔNG BAO GIỜ sửa source of truth!             │
  └─────────────────────────────────────────────────────────┘
```

```
REACT — MUTATION = BUG:
═══════════════════════════════════════════════════════════════

  ❌ BAD — mutation:
  ┌──────────────────────────────────────────────────────┐
  │ function handleSort() {                              │
  │   items.sort();        // MUTATE items!              │
  │   setItems(items);     // React: "Hm, same ref...   │
  │ }                      //  KHÔNG re-render!" 😱      │
  │                                                      │
  │ → React so sánh REFERENCE (===)                      │
  │ → items.sort() sửa tại chỗ → ref KHÔNG ĐỔI         │
  │ → React nghĩ state KHÔNG thay đổi → KHÔNG render!   │
  └──────────────────────────────────────────────────────┘

  ✅ GOOD — non-mutating:
  ┌──────────────────────────────────────────────────────┐
  │ function handleSort() {                              │
  │   setItems(items.toSorted()); // MỚI! ref KHÁC!     │
  │ }                                                    │
  │ // hoặc:                                             │
  │ setItems([...items].sort());  // spread + sort       │
  │                                                      │
  │ → Mảng mới = ref MỚI → React detect → RE-RENDER!   │
  └──────────────────────────────────────────────────────┘
```

## §3. ES2023: toReversed, toSorted, toSpliced — Non-Mutating!

### 3.1 Bài Viết: "Cứu Tinh" Đến Muộn 28 Năm

Will mỉa mai: _"Under 30 years — a really fast update!"_ — JS mất gần 28 năm (1995→2023) để thêm phiên bản non-mutating!

> _"Inside of toReversed, toSpliced and toSorted — the same has happened as in our copyArrayAndManipulate. We're creating a brand new array inside, putting changed elements, returned out."_

```
MUTATING vs NON-MUTATING — SO SÁNH:
═══════════════════════════════════════════════════════════════

  ❌ CŨ (MUTATING):           ✅ MỚI (NON-MUTATING):
  ┌──────────────────────┐    ┌──────────────────────────┐
  │ array.reverse()      │    │ array.toReversed()       │
  │ → sửa array GỐC     │    │ → trả mảng MỚI          │
  │ → return array gốc   │    │ → array GỐC KHÔNG ĐỔI   │
  ├──────────────────────┤    ├──────────────────────────┤
  │ array.sort()         │    │ array.toSorted()         │
  │ → sắp xếp TẠI CHỖ   │    │ → trả mảng MỚI đã sort │
  │ → return array gốc   │    │ → array GỐC KHÔNG ĐỔI   │
  ├──────────────────────┤    ├──────────────────────────┤
  │ array.splice(i,n,x)  │    │ array.toSpliced(i,n,x)  │
  │ → cắt/chèn TẠI CHỖ  │    │ → trả mảng MỚI đã splice│
  │ → return phần bị cắt │    │ → array GỐC KHÔNG ĐỔI   │
  └──────────────────────┘    └──────────────────────────┘

  → Prefix "to" = "biến thành" = TẠO MỚI, không sửa!
```

### 3.2 Code So Sánh

```javascript
// SO SÁNH: MUTATING vs NON-MUTATING

const array1 = [1, 2, 3];
const array2 = [1, 2, 3];

// ═══ MUTATING (array1 — bị phá huỷ!) ═══

array1.reverse(); // array1 = [3, 2, 1] ← BỊ ĐỔI!
array1.splice(1, 1, 6); // array1 = [3, 6, 1] ← BỊ ĐỔI!
array1.sort(); // array1 = [1, 3, 6] ← BỊ ĐỔI!
console.log(array1); // [1, 3, 6] ← gốc [1,2,3] BIẾN MẤT!

// ═══ NON-MUTATING (array2 — an toàn!) ═══

const reversed = array2.toReversed(); // [3, 2, 1]
const spliced = array2.toSpliced(1, 1, 6); // [1, 6, 3]
const sorted = array2.toSorted(); // [1, 2, 3]

console.log(array2); // [1, 2, 3] ← VẪN NGUYÊN VẸN! ✅
console.log(reversed); // [3, 2, 1] ← mảng MỚI
console.log(spliced); // [1, 6, 3] ← mảng MỚI
console.log(sorted); // [1, 2, 3] ← mảng MỚI
```

```
MEMORY — MUTATING vs NON-MUTATING:
═══════════════════════════════════════════════════════════════

  MUTATING (array1):

  HEAP:
  ┌──────────────────┐
  │ 0x7f2a: [1,2,3]  │ ← BAN ĐẦU
  │                   │
  │ .reverse()        │
  │ 0x7f2a: [3,2,1]  │ ← SỬA cùng vùng nhớ!
  │                   │
  │ .splice(1,1,6)    │
  │ 0x7f2a: [3,6,1]  │ ← LẠI SỬA!
  │                   │
  │ .sort()           │
  │ 0x7f2a: [1,3,6]  │ ← LẠI SỬA!
  └──────────────────┘
  → [1,2,3] gốc BIẾN MẤT từ memory!

  NON-MUTATING (array2):

  HEAP:
  ┌──────────────────────────────────────────┐
  │ 0x8a3b: [1,2,3]  ← array2 (KHÔNG ĐỔI!)  │
  │ 0x8a4c: [3,2,1]  ← reversed (MỚI!)      │
  │ 0x8a5d: [1,6,3]  ← spliced (MỚI!)       │
  │ 0x8a6e: [1,2,3]  ← sorted (MỚI!)        │
  └──────────────────────────────────────────┘
  → array2 NGUYÊN VẸN ở 0x8a3b!
  → Mỗi method tạo vùng nhớ MỚI!
```

## §4. Chaining Non-Mutating Methods

Will teaser: _"We could call on that toSpliced, skip saving individual labels, call on that toSorted, and each time our array2 would stay fixed."_

```javascript
// CHAINING — Mỗi bước tạo mảng MỚI → chain tiếp!

const array2 = [1, 2, 3];

const finalResult = array2
  .toReversed() // [3, 2, 1]
  .toSpliced(1, 1, 6) // [3, 6, 1]
  .toSorted(); // [1, 3, 6]

console.log(array2); // [1, 2, 3] ← VẪN NGUYÊN VẸN! ✅
console.log(finalResult); // [1, 3, 6] ← kết quả chain
```

```
CHAINING — PIPELINE:
═══════════════════════════════════════════════════════════════

  array2
    │ [1, 2, 3]
    ↓
  .toReversed()
    │ [3, 2, 1]     ← mảng MỚI #1
    ↓
  .toSpliced(1,1,6)
    │ [3, 6, 1]     ← mảng MỚI #2 (từ #1)
    ↓
  .toSorted()
    │ [1, 3, 6]     ← mảng MỚI #3 (từ #2)
    ↓
  finalResult = [1, 3, 6]

  → Mỗi bước: input = mảng trước đó
  → array2 KHÔNG BAO GIỜ bị sửa!
  → CHÍNH XÁC pattern copyArrayAndManipulate!
```

## §5. ES2019: Array.flat() — Flatten Nested Arrays

### 5.1 Bài Viết: "Làm Phẳng" Mảng Lồng Nhau

> Will: _"If we have an array containing another array inside of it, we can call the flat method, and that will take out the subarray and assign it as individual elements."_

```javascript
// Array.flat() — ES2019 — NON-MUTATING!

const deepArray = [1, 2, [1, 2], 2];

// Flat 1 level (default):
const flattened = deepArray.flat();
console.log(flattened); // [1, 2, 1, 2, 2]
console.log(deepArray); // [1, 2, [1, 2], 2] ← KHÔNG ĐỔI! ✅

// Flat nhiều levels:
const deeper = [1, [2, [3, [4, [5]]]]];
deeper.flat(1); // [1, 2, [3, [4, [5]]]]
deeper.flat(2); // [1, 2, 3, [4, [5]]]
deeper.flat(Infinity); // [1, 2, 3, 4, 5] ← flat HẾT!

// ⚠️ Will: "we can pass in Infinity — which is a terrible
//    name for it — it defaults to one"
```

```
FLAT() — TRỰC QUAN:
═══════════════════════════════════════════════════════════════

  Input:    [1, 2, [1, 2], 2]

  flat(1):  [1, 2,  1, 2,  2]    ← mở 1 tầng ngoặc
                ↑  ↑
              sub-array được "trải ra"

  Deeper:   [1, [2, [3, [4]]]]

  flat(1):  [1,  2, [3, [4]]]    ← mở 1 tầng
  flat(2):  [1,  2,  3, [4] ]    ← mở 2 tầng
  flat(3):  [1,  2,  3,  4  ]    ← mở 3 tầng
  flat(∞):  [1,  2,  3,  4  ]    ← mở HẾT!

  → depth = "bao nhiêu tầng muốn mở"
  → Default = 1
  → Infinity = mở tất cả
```

## §6. ES2023: findLastIndex() — Tìm Từ Cuối

```javascript
// findLastIndex() — ES2023 — tìm NGƯỢC từ cuối

const flattened = [1, 2, 1, 2, 2];

// findIndex: tìm XUÔI → trả index ĐẦU TIÊN match
flattened.findIndex((el) => el === 2); // 1  (vị trí 1)

// findLastIndex: tìm NGƯỢC → trả index CUỐI CÙNG match
flattened.findLastIndex((el) => el === 2); // 4  (vị trí 4)
```

```
findIndex vs findLastIndex:
═══════════════════════════════════════════════════════════════

  Array: [1, 2, 1, 2, 2]
  Index:  0  1  2  3  4

  findIndex(el => el === 2):
  → Duyệt XUÔI: 0→1→2→3→4
  → Gặp 2 ở index 1 → DỪNG → return 1

  findLastIndex(el => el === 2):
  → Duyệt NGƯỢC: 4→3→2→1→0
  → Gặp 2 ở index 4 → DỪNG → return 4

  Callback: el => el === 2
  → HOF (findLastIndex) nhận callback!
  → Return true → "tìm thấy!" → trả index
  → Return false → "chưa" → tiếp tục ngược
```

## §8. Tự Implement: Tất Cả Methods Từ Đầu

```javascript
// TỰ BUILD: toReversed — KHÔNG dùng built-in!

function myToReversed(array) {
  const output = []; // Mảng MỚI!
  for (let i = array.length - 1; i >= 0; i--) {
    output.push(array[i]);
  }
  return output; // Return mảng MỚI, gốc KHÔNG ĐỔI!
}

const arr = [1, 2, 3];
console.log(myToReversed(arr)); // [3, 2, 1]
console.log(arr); // [1, 2, 3] ← KHÔNG ĐỔI! ✅
```

```javascript
// TỰ BUILD: toSorted — KHÔNG dùng built-in!

function myToSorted(array, compareFn) {
  // Tạo COPY rồi sort copy (không sửa gốc)
  const copy = [];
  for (let i = 0; i < array.length; i++) {
    copy.push(array[i]);
  }

  // Simple bubble sort (chỉ để demo):
  for (let i = 0; i < copy.length; i++) {
    for (let j = 0; j < copy.length - 1 - i; j++) {
      const shouldSwap = compareFn
        ? compareFn(copy[j], copy[j + 1]) > 0
        : String(copy[j]) > String(copy[j + 1]);
      if (shouldSwap) {
        const temp = copy[j];
        copy[j] = copy[j + 1];
        copy[j + 1] = temp;
      }
    }
  }
  return copy;
}

const nums = [3, 1, 4, 1, 5];
console.log(myToSorted(nums)); // [1, 1, 3, 4, 5]
console.log(myToSorted(nums, (a, b) => b - a)); // [5, 4, 3, 1, 1]
console.log(nums); // [3, 1, 4, 1, 5] ← KHÔNG ĐỔI!
```

```javascript
// TỰ BUILD: toSpliced — KHÔNG dùng built-in!

function myToSpliced(array, start, deleteCount, ...items) {
  const output = [];

  // Copy phần TRƯỚC start:
  for (let i = 0; i < start; i++) {
    output.push(array[i]);
  }

  // Chèn items mới:
  for (const item of items) {
    output.push(item);
  }

  // Copy phần SAU (skip deleteCount elements):
  for (let i = start + deleteCount; i < array.length; i++) {
    output.push(array[i]);
  }

  return output;
}

const data = [1, 2, 3, 4, 5];
console.log(myToSpliced(data, 1, 2, 8, 9)); // [1, 8, 9, 4, 5]
console.log(data); // [1, 2, 3, 4, 5] ← KHÔNG ĐỔI!
```

```javascript
// TỰ BUILD: flat — KHÔNG dùng built-in!

function myFlat(array, depth = 1) {
  const output = [];

  for (let i = 0; i < array.length; i++) {
    if (Array.isArray(array[i]) && depth > 0) {
      // Recursively flatten sub-arrays:
      const flattened = myFlat(array[i], depth - 1);
      for (const item of flattened) {
        output.push(item);
      }
    } else {
      output.push(array[i]);
    }
  }

  return output;
}

console.log(myFlat([1, [2, [3, [4]]]])); // [1, 2, [3, [4]]]
console.log(myFlat([1, [2, [3, [4]]]], 2)); // [1, 2, 3, [4]]
console.log(myFlat([1, [2, [3, [4]]]], Infinity)); // [1, 2, 3, 4]
```

```javascript
// TỰ BUILD: findLastIndex — KHÔNG dùng built-in!

function myFindLastIndex(array, callback) {
  for (let i = array.length - 1; i >= 0; i--) {
    if (callback(array[i], i, array)) {
      return i; // Tìm thấy → trả index!
    }
  }
  return -1; // Không tìm thấy
}

console.log(myFindLastIndex([1, 2, 1, 2, 2], (el) => el === 2)); // 4
```

## §9. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 9.1 Pattern ①: 5 Whys — Tại Sao Cần Non-Mutating?

```
5 WHYS: TẠI SAO ES2023 THÊM toReversed/toSorted/toSpliced?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao cần phiên bản non-mutating?
  └→ Vì reverse/sort/splice SỬA mảng gốc = SIDE EFFECT!
     Trái với pure function principle.

  WHY ②: Tại sao side effect xấu cho array methods?
  └→ Vì arrays thường SHARED reference. Nếu ComponentA
     sort() mảng mà ComponentB cũng dùng → B BỊ ẢNH HƯỞNG
     mà KHÔNG BIẾT!

  WHY ③: Tại sao không sửa từ đầu (1995)?
  └→ Vì năm 1995, memory đắt → tạo mảng mới = LÃNG PHÍ.
     Mutation tại chỗ tiết kiệm RAM. Năm 2023, RAM rẻ,
     developer time ĐẮT hơn.

  WHY ④: Tại sao đợi đến 2023?
  └→ Vì backward compatibility! Nếu đổi reverse() thành
     non-mutating → TRIỆU websites bị break! Thay vào đó
     TC39 tạo METHOD MỚI: toReversed(). Cũ vẫn chạy.

  WHY ⑤: Tại sao prefix "to"?
  └→ "to" gợi ý CHUYỂN ĐỔI → kết quả MỚI.
     toString() = tạo string MỚI, KHÔNG sửa object.
     toReversed() = tạo array MỚI, KHÔNG sửa gốc.
     ⭐ Naming convention = tín hiệu cho developer!
```

### 9.2 Pattern ②: First Principles

```
FIRST PRINCIPLES — MUTATING vs NON-MUTATING:

  NON-MUTATING method = hàm TẠO output mới:
  ┌──────────────────────────────────────────────────────┐
  │ ① Nhận reference đến array trên heap                │
  │ ② ALLOCATE vùng nhớ MỚI trên heap                  │
  │ ③ Copy + transform elements vào vùng mới            │
  │ ④ Return reference ĐẾN VÙNG MỚI                    │
  │ → Vùng nhớ gốc KHÔNG bị sửa!                        │
  └──────────────────────────────────────────────────────┘

  CHÍNH XÁC = copyArrayAndManipulate pattern!
  → Tạo output = []
  → Duyệt + transform
  → Return output
```

```javascript
// TỰ BUILD: "SAFE" WRAPPER — Biến mutating → non-mutating

function makeSafe(mutatingMethod) {
  return function (array, ...args) {
    const copy = [...array]; // Copy TRƯỚC khi mutate!
    mutatingMethod.apply(copy, args);
    return copy; // Return bản copy đã mutate!
  };
}

// Biến reverse, sort, splice thành safe versions:

const safeSort = makeSafe(Array.prototype.sort);

console.log(safeSort(original)); // [1, 1, 3, 4, 5]
```
