## §1. Big Three — 3 Thành Phần Cốt Lõi Của JS Runtime

### 1.1 Bài Viết Chuyên Sâu

Will Sentance gọi đây là **"the last of the big three"** — ba thành phần tạo nên toàn bộ JavaScript runtime. Mọi thứ khác (closure, prototype, async) chỉ là **features xây trên** 3 nền tảng này:

1. **Thread of Execution** — Đọc và chạy code từng dòng
2. **Memory** — Lưu trữ dữ liệu (biến, hàm, objects)
3. **Call Stack** — Theo dõi function nào đang chạy, quay về đâu khi xong

Phần 1.1 đã dạy ① và ②. Bây giờ chúng ta hoàn thành bức tranh với ③ — **Call Stack**.

> **Tại sao Call Stack quan trọng?** Khi bạn gọi function A, bên trong A gọi B, bên trong B gọi C — JavaScript cần biết: "Tôi đang chạy C. Khi C xong, quay lại B. Khi B xong, quay lại A. Khi A xong, quay lại Global." Call Stack là **bộ nhớ thứ tự** giúp JS không bao giờ "lạc đường".

```
BIG THREE — 3 THÀNH PHẦN CỐT LÕI:
═══════════════════════════════════════════════════════════════

  ┌─────────────────────────────────────────────────────────┐
  │              JAVASCRIPT RUNTIME                         │
  │                                                         │
  │  ① THREAD OF EXECUTION        (Phần 1.1 đã học)        │
  │  → Đọc code từng dòng, synchronous, single-threaded    │
  │                                                         │
  │  ② MEMORY                     (Phần 1.1 đã học)        │
  │  → Lưu biến, hàm, objects vào bộ nhớ                   │
  │                                                         │
  │  ③ CALL STACK                 ← BÀI NÀY!               │
  │  → Theo dõi function đang chạy                          │
  │  → Biết quay về đâu khi function xong                   │
  │                                                         │
  └─────────────────────────────────────────────────────────┘

  Will Sentance tóm tắt:
  ① Do code       (Thread of Execution)
  ② Save stuff    (Memory)
  ③ Keep track    (Call Stack)

  "Everything else is a feature of all this,
   or adjacent to all this."
  → Closure, Prototype, Async = FEATURES xây trên Big Three!
```

### 1.2 Tại Sao JS Không Tự Biết Quay Về Đâu?

Câu hỏi hay mà Will đặt ra: **"JavaScript doesn't automatically know where to go back to."**

Khi thread "dệt vào" function, nó rời khỏi chỗ cũ. Khi function xong (return), thread cần biết: **"Quay lại dòng nào ở chỗ cũ?"** Nếu không có Call Stack, JS sẽ "quên" mình đang ở đâu — giống như bạn đọc sách, nhảy sang footnote, rồi quên đang đọc trang nào.

```
VẤN ĐỀ: KHÔNG CÓ CALL STACK
═══════════════════════════════════════════════════════════════

  function a() { b(); console.log("a done"); }
  function b() { c(); console.log("b done"); }
  function c() { console.log("c done"); }
  a();

  KHÔNG CÓ CALL STACK:
  Thread → vào a() → vào b() → vào c()
  c() xong... → quay về ĐÂU? b? a? Global? 😱
  → JS KHÔNG BIẾT! Cần MỘT THỨ để theo dõi!

  CÓ CALL STACK:
  ┌────────────┐
  │   c()      │ ← hiện tại ở đây
  ├────────────┤
  │   b()      │ ← c xong → quay lại đây
  ├────────────┤
  │   a()      │ ← b xong → quay lại đây
  ├────────────┤
  │  Global    │ ← a xong → quay lại đây
  └────────────┘
  → Nhìn Call Stack = biết NGAY quay về đâu!
```

## §2. Call Stack — JavaScript Biết Mình Đang Ở Đâu Nhờ Gì?

### 2.1 Bài Viết: Stack Là Gì?

**Stack** là một cấu trúc dữ liệu LIFO — **Last In, First Out** (vào sau, ra trước). Giống như một chồng đĩa:

- Đặt đĩa lên (**push**) → đĩa mới ở TRÊN cùng
- Lấy đĩa ra (**pop**) → luôn lấy đĩa TRÊN cùng
- Bạn KHÔNG THỂ lấy đĩa ở giữa!

Will Sentance nhấn mạnh: **"It only cares about what is on the very top of it."** — Call Stack chỉ quan tâm cái gì ở **TRÊN CÙNG**.

```
CALL STACK — CẤU TRÚC LIFO:
═══════════════════════════════════════════════════════════════

  PUSH (thêm vào):                POP (lấy ra):
  ┌────────────┐                  ┌────────────┐
  │   func3() ←│── push           │   func3() ─│──→ pop (xoá!)
  ├────────────┤                  ├────────────┤
  │   func2()  │                  │   func2() ←│── giờ ở trên cùng
  ├────────────┤                  ├────────────┤
  │   func1()  │                  │   func1()  │
  ├────────────┤                  ├────────────┤
  │   Global   │                  │   Global   │
  └────────────┘                  └────────────┘

  NGUYÊN TẮC:
  ─────────────
  • TOP of stack = function ĐANG CHẠY
  • Push = GỌI function mới → thêm lên đỉnh
  • Pop = function RETURN → xoá khỏi đỉnh
  • Sau pop → nhìn top → đó là nơi QUAY VỀ
  • Global LUÔN ở đáy → không bao giờ pop
    (trừ khi chương trình kết thúc hoàn toàn)
```

### 2.2 Khi Nào Push? Khi Nào Pop?

```
PUSH vs POP — NGUYÊN TẮC:
═══════════════════════════════════════════════════════════════

  PUSH (thêm vào Call Stack) khi:
  ┌──────────────────────────────────────────────────────┐
  │ ① Bắt đầu chạy program    → Push "Global"           │
  │ ② Gọi function (dấu ())   → Push tên function       │
  │                                                      │
  │ Ví dụ: multiplyBy2(3)     → Push "multiplyBy2"      │
  │         myFunc()           → Push "myFunc"            │
  └──────────────────────────────────────────────────────┘

  POP (xoá khỏi Call Stack) khi:
  ┌──────────────────────────────────────────────────────┐
  │ ① Function gặp "return"   → Pop function đó          │
  │ ② Function hết code       → Implicit return → Pop    │
  │ ③ Chương trình kết thúc   → Pop "Global"             │
  │                                                      │
  │ Sau pop → nhìn TOP        → đó là nơi tiếp tục!     │
  └──────────────────────────────────────────────────────┘

  ⚠️ KHÔNG POP khi:
  ─────────────────
  → Đang ở giữa function (chưa return)
  → Call Stack chỉ có Global (chương trình chưa xong)
```

## §4. Return — Keyword Thoát Khỏi Function

### 4.1 Bài Viết: Return = "Exit" + "Pop"

Will Sentance nhấn mạnh: **"Return — that's how JavaScript knows to exit out."** Keyword `return` làm 3 thứ cùng lúc:

1. **Evaluate** — Tính giá trị biểu thức sau `return`
2. **Exit** — Thoát khỏi Execution Context hiện tại
3. **Pop** — Xoá function khỏi Call Stack

Nếu không có `return`, function vẫn thoát khi hết code (implicit `return undefined`), nhưng explicit `return` cho phép bạn **kiểm soát** khi nào thoát và mang theo giá trị gì.

```
RETURN = 3 HÀNH ĐỘNG ĐỒNG THỜI:
═══════════════════════════════════════════════════════════════

  return result;

  ① EVALUATE:   result → tìm trong local memory → 6
  ② EXIT:       Thoát Execution Context → local memory bị XOÁ
  ③ POP:        Xoá function khỏi Call Stack
                → TOP of stack = nơi quay về

  ┌──────────────────────────────────────────────────────┐
  │  TRƯỚC return:           │  SAU return:              │
  │                          │                           │
  │  Call Stack:             │  Call Stack:              │
  │  ┌──────────────┐       │  ┌──────────────┐        │
  │  │ multiplyBy2 ←│ top   │  │              │        │
  │  ├──────────────┤       │  │              │        │
  │  │ Global()     │       │  │ Global() ←───│ top    │
  │  └──────────────┘       │  └──────────────┘        │
  │                          │                           │
  │  EC: inputNumber=3       │  EC: bị XOÁ!             │
  │      result=6            │  output = 6 (trả ra)     │
  └──────────────────────────┴───────────────────────────┘
```

## §5. Global Luôn Ở Đáy — Không Bao Giờ Bị Pop

### 5.1 Bài Viết: Global = "main()" Của JavaScript

Trong Java/C, bạn phải viết `main()` — entry point của chương trình. Trong JavaScript, **Global Execution Context đóng vai trò đó** — nó được tạo TỰ ĐỘNG khi bạn chạy file `.js`.

Will nói: _"We don't call it, we don't say 'run global', we just start running JavaScript."_

```
GLOBAL = ĐÁY CỦA CALL STACK:
═══════════════════════════════════════════════════════════════

  Java:                          JavaScript:
  ┌──────────────────────┐      ┌──────────────────────┐
  │ public static void   │      │ // Không cần "main"! │
  │ main(String[] args) {│      │ // Code chạy NGAY    │
  │   // entry point     │      │ const x = 5;         │
  │ }                    │      │ doStuff();            │
  └──────────────────────┘      └──────────────────────┘
  → Phải khai báo main()        → Global EC = main() ẩn

  KHI NÀO GLOBAL BỊ POP?
  ─────────────────────────
  ① Chương trình hết code → pop Global → Call Stack TRỐNG
  ② NHƯNG nếu có setTimeout, Promises, event listeners...
     → Global CHƯA pop! Phải đợi KHÔNG CÒN GÌ chờ chạy
     → (Xem phần Async/Event Loop)

  ⚠️ Will teaser: "Imagine if there were things that could
     come back in and arbitrarily start running in my code."
     → Đó là ASYNC! Call Stack + Event Loop = phần 5!
```

## §6. Function: Define Once, Use Many Times

### 6.1 Bài Viết: DRY Principle Nguyên Thuỷ

Will hỏi: _"How many times do we define our function? One time. And how many times can we reuse it? As much memory as you have."_

Đây là **DRY (Don't Repeat Yourself)** ở mức cơ bản nhất — function cho phép bạn viết code 1 lần rồi gọi bao nhiêu lần tuỳ thích. Mỗi lần gọi = Execution Context MỚI + push lên Call Stack.

```
DEFINE ONCE, USE MANY:
═══════════════════════════════════════════════════════════════

  DEFINE (1 lần):                 USE (nhiều lần):
  ┌──────────────────────┐       ┌──────────────────────┐
  │ function multiply(x) │       │ multiply(3)  → EC #1 │
  │ {                    │       │ multiply(10) → EC #2 │
  │   return x * 2;      │  ──→  │ multiply(7)  → EC #3 │
  │ }                    │       │ multiply(0)  → EC #4 │
  │                      │       │ ...          → EC #N │
  └──────────────────────┘       └──────────────────────┘
    Lưu vào memory 1 lần           Mỗi lần gọi = context riêng

  Call Stack sẽ lần lượt:
  ① [Global, multiply(3)]   → pop → [Global]
  ② [Global, multiply(10)]  → pop → [Global]
  ③ [Global, multiply(7)]   → pop → [Global]
  → Cùng function, nhưng mỗi lần = context HOÀN TOÀN MỚI!
```

## §7. Lưu Trữ Trong Memory: Trực Tiếp vs Reference

### 7.1 Bài Viết: Teaser Về Primitives vs Objects

Will đề cập một insight quan trọng (sẽ học sâu ở phần type coercion): **Một số giá trị được lưu TRỰC TIẾP trong memory, một số được lưu dưới dạng REFERENCE (con trỏ).**

```
TRỰC TIẾP vs REFERENCE:
═══════════════════════════════════════════════════════════════

  TRỰC TIẾP (Primitives):
  ┌──────────────────────────────────────────────────────┐
  │ Kiểu: number, string, boolean, null, undefined      │
  │                                                      │
  │ Memory:                                              │
  │   num ──→ [ 3 ]         ← GIÁ TRỊ nằm ngay ở đây!  │
  │   name ─→ [ "Jun" ]     ← GIÁ TRỊ nằm ngay ở đây!  │
  │   ok ───→ [ true ]      ← GIÁ TRỊ nằm ngay ở đây!  │
  │                                                      │
  │ → Copy = tạo BẢN SAO ĐỘC LẬP                       │
  │   let a = 3; let b = a; b = 5;                      │
  │   → a vẫn = 3! (b là bản sao riêng)                 │
  └──────────────────────────────────────────────────────┘

  REFERENCE (Objects, Arrays, Functions):
  ┌──────────────────────────────────────────────────────┐
  │ Kiểu: object, array, function                       │
  │                                                      │
  │ Memory:                                              │
  │   arr ──→ [ 0x7f2a ] ──→ Heap: [1, 2, 3]            │
  │             ↑ pointer       ↑ giá trị thực ở ĐÂY    │
  │   fn  ──→ [ 0x8b3c ] ──→ Heap: function code...     │
  │                                                      │
  │ → Copy = sao chép POINTER, không phải giá trị!       │
  │   let a = [1,2]; let b = a; b.push(3);              │
  │   → a = [1,2,3]! (cùng trỏ đến 1 mảng!)            │
  └──────────────────────────────────────────────────────┘

  Will nói: "This will have consequences in our type
  coercion section." → Sẽ học sâu ở phần sau!
```

## §8. Tự Implement: Call Stack Engine

```javascript
// TỰ XÂY LẠI CALL STACK — Không dùng thư viện!

class CallStack {
  constructor(maxSize = 10000) {
    this.frames = [];
    this.maxSize = maxSize;
  }

  // Push: Khi GỌI function
  push(frameName) {
    if (this.frames.length >= this.maxSize) {
      throw new RangeError(
        `Maximum call stack size exceeded! (${this.maxSize} frames)`,
      );
    }
    this.frames.push(frameName);
    console.log(`[PUSH] ${frameName}`);
    this.print();
  }

  // Pop: Khi function RETURN
  pop() {
    if (this.frames.length === 0) {
      throw new Error("Cannot pop empty call stack!");
    }
    const removed = this.frames.pop();
    console.log(`[POP]  ${removed}`);
    this.print();
    return removed;
  }

  // Top: Function đang chạy
  top() {
    return this.frames[this.frames.length - 1] || null;
  }

  // In Call Stack hiện tại
  print() {
    if (this.frames.length === 0) {
      console.log("  Stack: [EMPTY]");
      return;
    }
    const visual = this.frames
      .map((f, i) => {
        const marker = i === this.frames.length - 1 ? " ← running" : "";
        return `  │ ${f}${marker} │`;
      })
      .reverse()
      .join("\n");
    console.log(`  ┌${"─".repeat(30)}┐\n${visual}\n  └${"─".repeat(30)}┘`);
  }
}

// MÔ PHỎNG CODE TỪ BÀI HỌC:
const stack = new CallStack();

// Bật JavaScript → Push Global
stack.push("Global()");

// const num = 3; → chỉ lưu memory, KHÔNG push
console.log("\n--- const num = 3 (no stack change) ---");

// const output = multiplyBy2(num); → PUSH!
console.log("\n--- Calling multiplyBy2(3) ---");
stack.push("multiplyBy2(3)");

// return result; → POP!
console.log("\n--- return 6 ---");
stack.pop(); // pop multiplyBy2(3)

// const newOutput = multiplyBy2(10); → PUSH!
console.log("\n--- Calling multiplyBy2(10) ---");
stack.push("multiplyBy2(10)");

// return result; → POP!
console.log("\n--- return 20 ---");
stack.pop(); // pop multiplyBy2(10)

// Chương trình kết thúc → POP Global
console.log("\n--- Program ends ---");
stack.pop(); // pop Global()
// Stack: [EMPTY] → chương trình hoàn toàn kết thúc!
```

```javascript
// NÂNG CAO: MÔ PHỎNG ĐỆ QUY VÀ STACK OVERFLOW

class CallStackWithDepthTracker extends CallStack {
  constructor(maxSize = 15) {
    // Giới hạn nhỏ để demo
    super(maxSize);
    this.maxDepthReached = 0;
  }

  push(frameName) {
    super.push(frameName);
    if (this.frames.length > this.maxDepthReached) {
      this.maxDepthReached = this.frames.length;
    }
  }
}

// Demo: đệ quy factorial
const demoStack = new CallStackWithDepthTracker(8);
demoStack.push("Global()");

function simulateFactorial(n) {
  demoStack.push(`factorial(${n})`);

  if (n <= 1) {
    console.log(`  → Base case! Return 1`);
    demoStack.pop();
    return 1;
  }

  const result = n * simulateFactorial(n - 1);
  console.log(`  → Return ${result}`);
  demoStack.pop();
  return result;
}

try {
  console.log("\n=== Factorial(5) ===");
  const answer = simulateFactorial(5);
  console.log(`\nAnswer: ${answer}`);
  console.log(`Max depth: ${demoStack.maxDepthReached}`);
} catch (e) {
  console.error(`💥 ${e.message}`);
}

// Max depth = 6 (Global + factorial(5) + ... + factorial(1))
// Nếu gọi factorial(100) với maxSize=8 → Stack Overflow! 💥
```

## §9. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 9.1 Pattern ①: 5 Whys — Call Stack

```
5 WHYS: TẠI SAO CẦN CALL STACK?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao JS cần Call Stack?
  └→ Vì khi function gọi function, JS cần biết
     QUAY VỀ ĐÂU khi function con xong.

  WHY ②: Tại sao không lưu "return address" trong biến?
  └→ Vì function có thể gọi NHIỀU function lồng nhau:
     a() → b() → c() → d()... Cần danh sách có thứ tự!

  WHY ③: Tại sao dùng STACK (LIFO) chứ không phải Queue?
  └→ Vì function call LỒNG NHAU = function CUỐI gọi phải
     xong TRƯỚC (LIFO). Giống mở ngoặc: (( )) — đóng trong
     trước, đóng ngoài sau.

  WHY ④: Tại sao Stack có giới hạn kích thước?
  └→ Vì Stack nằm trên RAM vật lý. RAM = hữu hạn (1-8MB
     cho stack). Đệ quy vô hạn → vượt giới hạn → crash!

  WHY ⑤: Tại sao Stack Overflow gây crash, không tự mở rộng?
  └→ Vì Stack phải CONTIGUOUS (liên tục) trong RAM để
     CPU access nhanh (cache-friendly). Mở rộng = relocate
     toàn bộ = cực chậm + fragmentation. Trade-off:
     tốc độ (contiguous) ↔ giới hạn kích thước.
```

### 9.2 Pattern ②: First Principles

```
FIRST PRINCIPLES — PHÂN RÃ CALL STACK:
═══════════════════════════════════════════════════════════════

  DATA STRUCTURE: Stack (Array + LIFO constraint)
  ─────────────────────────────────────────────────
  • push() = O(1) — thêm vào cuối array
  • pop()  = O(1) — xoá từ cuối array
  • top()  = O(1) — đọc cuối array
  → Tất cả operations = O(1)! Cực nhanh!

  ALGORITHM:
  ─────────────────────────────────────────────────
  • Function call → push frame
  • Return        → pop frame + nhảy về top
  • Kiểm tra overflow trước mỗi push

  HARDWARE:
  ─────────────────────────────────────────────────
  • CPU có Stack Pointer register (SP/RSP)
  • Push = giảm SP, viết data vào [SP]
  • Pop  = đọc data từ [SP], tăng SP
  • Stack memory = contiguous → CPU cache HIT cao
  → Call Stack NHANH vì hardware HỖ TRỢ trực tiếp!
```

### 9.3 Pattern ③: Trade-off Analysis

```
TRADE-OFF: CALL STACK DESIGN
═══════════════════════════════════════════════════════════════

  ┌─────────────────┬──────────────────┬─────────────────────┐
  │ Quyết định       │ Lợi ích          │ Cái giá            │
  ├─────────────────┼──────────────────┼─────────────────────┤
  │ LIFO order      │ ✅ Nested calls  │ ❌ Không parallel   │
  │                 │   tự nhiên       │   execution         │
  ├─────────────────┼──────────────────┼─────────────────────┤
  │ Fixed size      │ ✅ Nhanh (cache) │ ❌ Stack Overflow   │
  │ (~1-8MB)        │ ✅ Predictable   │   khi đệ quy sâu   │
  ├─────────────────┼──────────────────┼─────────────────────┤
  │ Auto-pop on     │ ✅ Memory free   │ ❌ Data mất khi     │
  │ return          │   tự động        │   return (trừ close) │
  ├─────────────────┼──────────────────┼─────────────────────┤
  │ Single stack    │ ✅ No race cond. │ ❌ 1 blocking call  │
  │ (single-thread) │ ✅ Deterministic │   = freeze toàn bộ  │
  └─────────────────┴──────────────────┴─────────────────────┘

  KHI NÀO CALL STACK THẤT BẠI:
  → Đệ quy 10,000+ levels → Stack Overflow!
  → Fix: Tail Call Optimiz. (Safari), trampoline, iteration
  → Sync code chạy > 50ms → Block UI!
  → Fix: Web Workers, requestIdleCallback, chunking
```

### 9.5 Pattern ⑤: Reverse Engineering

// TỰ XÂY LẠI: STACK TRACE TỪ CALL STACK

function generateStackTrace(callStack) {
  // Stack trace = đọc Call Stack từ TRÊN xuống
  const trace = callStack.frames
    .slice()
    .reverse()
    .map((frame, i) => {
      const prefix = i === 0 ? "    at" : "    at";
      return `${prefix} ${frame}`;
    });

  return `Error: Something went wrong!\n${trace.join("\n")}`;
}

// Mô phỏng:
const stack = {
  frames: ["Global()", "processData()", "validate()", "checkEmail()"],
};
console.log(generateStackTrace(stack));
// Error: Something went wrong!
//     at checkEmail()       ← lỗi ở đây!
//     at validate()         ← gọi checkEmail
//     at processData()      ← gọi validate
//     at Global()           ← entry point

// → ĐỌC TỪ TRÊN XUỐNG = tìm lỗi!
// → ĐỌC TỪ DƯỚI LÊN = thứ tự gọi!
// → Chính xác cách Chrome DevTools hiện! ✅

### 9.6 Pattern ⑥: Lịch Sử & Tiến Hoá

```
LỊCH SỬ: CALL STACK TỪ HARDWARE ĐẾN JAVASCRIPT
═══════════════════════════════════════════════════════════════

  1945: Von Neumann Architecture
  │     → CPU có Program Counter (PC) = biết dòng nào đang chạy
  │     → CHƯA có stack → subroutine call = nightmare!
  │
  ↓
  1960s: Hardware Stack Support
  │     → CPU thêm Stack Pointer register (SP)
  │     → CALL instruction = push return address + jump
  │     → RET instruction = pop return address + jump back
  │     → Vấn đề: stack size cố định, không có GC
  │
  ↓
  1970s: C Language — Stack Frames
  │     → Mỗi function call = 1 stack frame
  │     → Frame chứa: locals, args, return address
  │     → Stack Overflow = crash → developer phải tự quản lý
  │
  ↓
  1995: JavaScript — Managed Call Stack
  │     → Engine tự quản lý Call Stack
  │     → Developer KHÔNG thấy stack trực tiếp
  │     → Nhưng RangeError = stack overflow = GIỐNG C!
  │
  ↓
  2009: V8 Engine — Optimized Stack
  │     → JIT compiler tối ưu stack frames
  │     → Inlining: "xoá" frame bằng cách inline function
  │     → Tail Call Optimization (ES6, chỉ Safari implement)
  │
  ↓
  2020+: DevTools Integration
        → Async stack traces (theo dõi qua promises)
        → Source maps (stack trace → original source)
        → Performance profiling dựa trên call stack

  ⭐ Call Stack JS = Hardware Stack (1960s) + Managed Runtime!
```
