## §1. Tổng Kết Lại: Function Có 2 Loại Bộ Nhớ

### 1.1 Bài Viết: "Two Types of Memory"

Will tổng kết closure bằng một câu cực kỳ chính xác:

> _"Suddenly my regular old functions have TWO TYPES of memory. The one they create LIVE when they run, and the one that STICKS AROUND in between invocations, and is only available to it."_

```
HAI LOẠI BỘ NHỚ CỦA FUNCTION:
═══════════════════════════════════════════════════════════════

  ① TEMPORARY MEMORY (Local Memory / Variable Environment):
  ┌──────────────────────────────────────────────────────┐
  │ → Được TẠO MỚI mỗi lần function chạy                │
  │ → BỊ XOÁ khi function kết thúc                       │
  │ → Chứa: parameters, local variables, inner functions │
  │ → "The one they create live when they run"            │
  │ → KHÔNG nhớ gì từ lần chạy trước!                    │
  └──────────────────────────────────────────────────────┘

  ② PERSISTENT MEMORY (Backpack / [[Environment]] / Closure):
  ┌──────────────────────────────────────────────────────┐
  │ → Được TẠO khi function ĐƯỢC KHAI BÁO (define time) │
  │ → TỒN TẠI giữa các lần gọi (invocations)            │
  │ → CHỈ function đó truy cập được (private!)          │
  │ → Chứa: biến từ enclosing scope mà function DÙNG   │
  │ → "The one that sticks around between invocations"   │
  │ → "And is ONLY AVAILABLE to it"                      │
  └──────────────────────────────────────────────────────┘

  SO SÁNH:
  ┌──────────────────┬──────────────────┬──────────────────┐
  │                  │ Temporary        │ Persistent       │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Tạo khi          │ Function CHẠY    │ Function DEFINE  │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Xoá khi          │ Function KẾT THÚC│ KHÔNG bao giờ*  │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Mỗi lần gọi      │ MỚI hoàn toàn   │ CÙNG data cũ    │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Ai access         │ Chính function   │ CHỈ function đó │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Tương đương       │ RAM (volatile)   │ SSD (persistent)│
  └──────────────────┴──────────────────┴──────────────────┘
  * Trừ khi function bị garbage collected (mất reference)

  Will: "This ain't global. This is data that is created,
  this is a memory that is only available to THAT function.
  But unlike the one that gets rewritten every time,
  THIS ONE STICKS AROUND."
```

## §2. Joe's Question: Backpack Chứa TẤT CẢ Hay Chỉ Biến Referenced?

### 2.1 Bài Viết: Câu Hỏi Sắc Bén Từ Joe

Joe hỏi một câu cực hay:

> Joe: _"When we return add1 from outer, is it returning EVERYTHING that's a sibling to add1? Like if there were other functions in that body? Or is it only keeping track of data that is actually referenced inside of that function?"_

Will trả lời rõ ràng:

> Will: _"JavaScript smartly, LITERALLY LOOKS at the body of the add1 function and goes: what identifiers, what labels do you ever make reference to? If they're not referenced in that function, when that function's returned out, they are what they call GARBAGE COLLECTED."_

```
SELECTIVE CLOSURE — CHỈ GIỮ CÁI CẦN:
═══════════════════════════════════════════════════════════════

  function outer() {
    let counter = 0;           ← USED bởi add1 ✅
    let anotherCounter = 5;    ← NOT used bởi add1 ❌
    let bigArray = [1,2,...];  ← NOT used bởi add1 ❌
    let config = { key: "x" }; ← NOT used bởi add1 ❌

    function add1() {
      counter++;               ← CHỈ dùng "counter"!
    }
    return add1;
  }

  const newFunc = outer();

  BACKPACK CỦA newFunc 🎒:
  ┌──────────────────────────────────────────────────────┐
  │ ✅ counter: 0        ← add1 DÙNG → GIỮ!            │
  │                                                      │
  │ ❌ anotherCounter    ← add1 KHÔNG dùng → XOÁ!       │
  │ ❌ bigArray          ← add1 KHÔNG dùng → XOÁ!       │
  │ ❌ config            ← add1 KHÔNG dùng → XOÁ!       │
  │                                                      │
  │ Will: "You can have 100 other things in here.        │
  │ If they're not referenced in that function, when     │
  │ that function's returned out, they are GARBAGE        │
  │ COLLECTED. Removed, deleted, forgotten about."        │
  └──────────────────────────────────────────────────────┘

  TẠI SAO?
  → "The only place we can ever access this data is by
    the very function that we've returned out."
  → Nếu add1 KHÔNG BAO GIỜ dùng anotherCounter
  → Thì KHÔNG CÓ CÁCH NÀO truy cập anotherCounter nữa
  → Nó thành "rác" → GC thu hồi → tiết kiệm memory!
```

## §3. Garbage Collection — JS Thông Minh, Chỉ Giữ Cái CẦN

```
GARBAGE COLLECTION + CLOSURE:
═══════════════════════════════════════════════════════════════

  V8 Engine tối ưu closure bằng STATIC ANALYSIS:

  ① Khi DEFINE inner function:
     → V8 PHÂN TÍCH body của inner function
     → Liệt kê tất cả identifiers (tên biến) được reference
     → Chỉ giữ REFERENCE đến các biến đó

  ② Khi RETURN inner ra + outer EC bị xoá:
     → Biến REFERENCED → SỐNG (trong backpack)
     → Biến NOT referenced → GC thu hồi (xoá!)

  ③ Ví dụ thực tế:

  function createHandler() {
    const smallData = 42;           // 8 bytes
    const hugeData = new Array(1e6); // ~8MB!

    return function handler() {
      return smallData;  // CHỈ dùng smallData!
    };
  }

  const fn = createHandler();
  // Backpack: { smallData: 42 }      ← 8 bytes!
  // hugeData (8MB) → GARBAGE COLLECTED! ← Tiết kiệm!

  → V8 KHÔNG giữ 8MB vô nghĩa!
  → Chỉ giữ 8 bytes cần thiết!
  → Closure KHÔNG lãng phí memory (khi dùng đúng)!

  Will: "JavaScript can do a little read of the details
  of the add1 function and go: I'm not going to save
  everything, there's no point. Only save the stuff
  that is RELEVANT."
```

## §4. Michael's Summary: "Dependency = Keep Reference"

Michael tóm tắt hoàn hảo:

> Michael: _"If it has a DEPENDENCY on something that is outside of it, it will keep a reference to that memory."_

```
MICHAEL'S RULE — DEPENDENCY = KEEP:
═══════════════════════════════════════════════════════════════

  function outer() {
    let a = 1;     // ← inner DÙNG a → KEEP ✅
    let b = 2;     // ← inner DÙNG b → KEEP ✅
    let c = 3;     // ← inner KHÔNG dùng c → GC ❌

    function inner() {
      return a + b; // DEPENDENCIES: a, b
    }               // c = KHÔNG phải dependency
    return inner;
  }

  QUY TẮC:
  ┌──────────────────────────────────────────────────────┐
  │ Inner function DÙNG biến ngoài? → DEPENDENCY!        │
  │ → GIỮ trong backpack!                                │
  │                                                      │
  │ Inner function KHÔNG dùng? → NOT dependency!         │
  │ → GARBAGE COLLECTED!                                 │
  │                                                      │
  │ Will: "That's exactly right."                        │
  └──────────────────────────────────────────────────────┘
```

## §5. Nested Closures — Chain Nhiều Tầng

Joe cũng hỏi về nested closures — function trả ra từ nhiều tầng:

> Will: _"This is a chain. Like many of these things, you're going to have a chain up. And so the answer is YES."_

```
NESTED CLOSURES — CHAIN NHIỀU TẦNG:
═══════════════════════════════════════════════════════════════

  function grandparent() {
    let x = "grand";

    function parent() {
      let y = "parent";

      function child() {
        return x + " + " + y; // DÙNG cả x (grand) VÀ y (parent)!
      }
      return child;
    }
    return parent;
  }

  const getChild = grandparent(); // → parent function
  const child = getChild();       // → child function
  child();  // "grand + parent"

  BACKPACK CHAIN:
  ┌──────────────────────────────────────────────────────┐
  │ child: ƒ                                            │
  │                                                      │
  │ 🎒 Closure (parent):                                 │
  │   y: "parent"                                        │
  │                                                      │
  │ 🎒🎒 Closure (grandparent):                           │
  │   x: "grand"                                         │
  │                                                      │
  │ Lookup: local → parent backpack → grandparent → global│
  └──────────────────────────────────────────────────────┘

  → child "nhớ" data từ CẢ parent VÀ grandparent!
  → Scope chain = chain of [[Environment]] references!
```

## §6. Async Callbacks — Tại Sao Backpack Đặc Biệt Quan Trọng

### 6.1 Bài Viết: Backpack Trong Thế Giới Bất Đồng Bộ

> Will: _"Especially if this function is being called WAY LATER because you've asynchronously brought it back in to be called when some data from an API returns — the ability for that function to hold onto its surrounding initial backpack of data is a REALLY POWERFUL thing, especially in an ASYNCHRONOUS language."_

```
ASYNC + CLOSURE — TẠI SAO QUAN TRỌNG:
═══════════════════════════════════════════════════════════════

  function fetchUserProfile(userId) {
    const requestUrl = `/api/users/${userId}`;
    const startTime = Date.now();

    // Callback này sẽ chạy SAU vài giây!
    fetch(requestUrl)
      .then(response => response.json())
      .then(userData => {
        // 2 GIÂY SAU...
        // fetchUserProfile đã CHẠY XONG TỪ LÂU!
        // EC đã BỊ XOÁ!
        // NHƯNG...

        console.log(`Got ${userData.name}`);
        console.log(`From: ${requestUrl}`);   // ← BACKPACK! 🎒
        console.log(`Took: ${Date.now() - startTime}ms`); // ← BACKPACK!

        // requestUrl, startTime = CLOSURE VARIABLES!
        // Chúng vẫn CÒN nhờ backpack!
        // Dù fetchUserProfile đã kết thúc từ lâu!
      });
  }

  fetchUserProfile(42);
  // → fetchUserProfile CHẠY XONG ngay lập tức!
  // → NHƯNG callback .then() chạy SAU 2 giây!
  // → Lúc đó, callback CẦN requestUrl, startTime!
  // → NHỜ CLOSURE → backpack vẫn giữ data!
  // → Không có closure → callback KHÔNG CÓ data! 💥

  ┌──────────────────────────────────────────────────────┐
  │ TIMELINE:                                            │
  │                                                      │
  │ t=0ms:   fetchUserProfile() chạy                     │
  │          → requestUrl = "/api/users/42"              │
  │          → startTime = 1709769600000                 │
  │          → fetch() gửi request                       │
  │          → fetchUserProfile() KẾT THÚC! EC XOÁ!     │
  │                                                      │
  │ t=2000ms: Response từ server đến!                    │
  │          → callback .then() chạy                     │
  │          → CẦN requestUrl → BACKPACK! ✅             │
  │          → CẦN startTime → BACKPACK! ✅              │
  │          → Closure = CỨU TINH! 🎒                    │
  └──────────────────────────────────────────────────────┘

  Will: "In its pure core, JavaScript is: do this, then
  this, and when you hit running a function, do all the
  stuff inside and then FORGET ABOUT IT, besides the
  return value. And yet it doesn't — because if you return
  a function out, that function gets this special BACKPACK."
```

## §7. Module Pattern Teaser — Private Data Trong Codebase Lớn

> Will: _"Think about the module pattern — you don't want in a 3 million line codebase anyone using the word 'result', because your colleague from 17 years ago used the label 'result'. You want to have labels like 'result' because they're useful, but not have it POLLUTE the global namespace."_

```
MODULE PATTERN = CLOSURE APPLICATION:
═══════════════════════════════════════════════════════════════

  VẤN ĐỀ: 3 triệu dòng code, 50 developers, 17 năm:

  ❌ KHÔNG CÓ CLOSURE:
  ┌──────────────────────────────────────────────────────┐
  │ // Developer A (2007):                               │
  │ var result = processPayment();                       │
  │                                                      │
  │ // Developer B (2024):                               │
  │ var result = fetchUserProfile(); // 💥 OVERRIDE!     │
  │                                                      │
  │ // Developer A's code BỊ HỎNG!                       │
  │ // 100 files khác cũng dùng "result" → CHAOS!       │
  └──────────────────────────────────────────────────────┘

  ✅ VỚI CLOSURE (Module Pattern):
  ┌──────────────────────────────────────────────────────┐
  │ const paymentModule = (function() {                  │
  │   let result = null;     // ← PRIVATE! Trong backpack│
  │   let transactions = []; // ← PRIVATE!              │
  │                                                      │
  │   return {                                           │
  │     process(amount) {                                │
  │       result = { amount, status: "done" };           │
  │       transactions.push(result);                     │
  │       return result;                                 │
  │     },                                               │
  │     getHistory() {                                   │
  │       return [...transactions];                      │
  │     }                                                │
  │   };                                                 │
  │ })();                                                │
  │                                                      │
  │ // "result" HOÀN TOÀN private!                       │
  │ // Không ai conflict được!                           │
  │ // NHƯNG vẫn dùng được qua public methods!          │
  └──────────────────────────────────────────────────────┘

  Will: "You bet you know how it's done — in vast,
  complex backpacks."
```

## §8. Có Thể Truy Cập Backpack Thủ Công?

Will đặt câu hỏi mà nhiều người thắc mắc:

> Will: _"Can I get MANUAL ACCESS to this data? Maybe I could go newFunc.backpack.counter. Maybe."_

```
CÓ THỂ TRUY CẬP BACKPACK THỦ CÔNG?
═══════════════════════════════════════════════════════════════

  function outer() {
    let counter = 0;
    return function() { counter++; return counter; };
  }
  const newFunc = outer();

  ❌ newFunc.backpack           → undefined
  ❌ newFunc.counter            → undefined
  ❌ newFunc.[[Environment]]    → SyntaxError!
  ❌ newFunc["[[Environment]]"] → undefined

  → Backpack = [[hidden property]] → KHÔNG truy cập code!
  → Đó là lý do gọi là [[double bracket]] = internal slot!
  → CHỈ JS engine truy cập, KHÔNG phải developer!

  ✅ CÁCH DUY NHẤT truy cập backpack:
  → GỌI function! newFunc() → counter++ → 1
  → Đó là TOÀN BỘ ý nghĩa của "private data"!

  ✅ CÁCH INSPECT (debugging only):
  → console.dir(newFunc) → [[Scopes]] → Closure(outer)
  → Chrome DevTools → Sources → Scope panel
  → NHƯNG đây là debugging tools, KHÔNG phải code access!

  ⭐ Closure = private data THẬT SỰ!
  → Giống private fields trong class (#field)
  → KHÔNG AI bên ngoài truy cập được!
  → Chỉ function có backpack = truy cập qua việc CHẠY!
```

## §9. Tự Implement: Chứng Minh Selective Closure & Nested Chain

```javascript
// CHỨNG MINH 1: Selective Closure — V8 chỉ giữ biến referenced

function selectiveTest() {
  let kept = "I survive!";
  let discarded = "I'm garbage collected!";
  let alsoDiscarded = new Array(1000000).fill("waste");

  return function () {
    return kept; // CHỈ reference "kept"
  };
}

const fn = selectiveTest();
console.dir(fn);
// [[Scopes]] → Closure (selectiveTest):
//   kept: "I survive!"
//   ← discarded KHÔNG CÓ!
//   ← alsoDiscarded (4MB) KHÔNG CÓ! → Tiết kiệm memory!

fn(); // "I survive!"
```

```javascript
// CHỨNG MINH 4: Async closure — data sống sót sau function kết thúc

function createDelayedLogger(label) {
  const createdAt = new Date().toISOString();
  let callCount = 0;

  return function log(message) {
    callCount++;
    console.log(`[${label}][#${callCount}][${createdAt}] ${message}`);
    // label, createdAt, callCount = TẤT CẢ từ backpack!
  };
}

const apiLogger = createDelayedLogger("API");
// createDelayedLogger ĐÃ CHẠY XONG!
// NHƯNG label, createdAt, callCount VẪN SỐNG!

setTimeout(() => apiLogger("Request sent"), 1000);
// 1s SAU: [API][#1][2026-03-07T...] Request sent

setTimeout(() => apiLogger("Response received"), 3000);
// 3s SAU: [API][#2][2026-03-07T...] Response received
// callCount = 2 → PERSISTENT! Nhớ lần gọi trước!
```

```javascript
// CHỨNG MINH 5: Module pattern — private + public hoàn chỉnh

const userModule = (function () {
  // === PRIVATE (backpack) ===
  let users = [];
  let nextId = 1;

  function validate(name) {
    if (!name || name.trim().length === 0) {
      throw new Error("Name cannot be empty!");
    }
  }

  // === PUBLIC (returned object) ===
  return {
    add(name) {
      validate(name);
      const user = { id: nextId++, name: name.trim() };
      users.push(user);
      return user;
    },
    getAll() {
      return [...users]; // Return COPY, not reference!
    },
    findById(id) {
      return users.find((u) => u.id === id) || null;
    },
    get count() {
      return users.length;
    },
  };
})();

userModule.add("Will");
userModule.add("Joe");
console.log(userModule.getAll()); // [{id:1,name:"Will"}, {id:2,name:"Joe"}]
console.log(userModule.count); // 2

// ❌ PRIVATE — không truy cập được!
console.log(userModule.users); // undefined
console.log(userModule.nextId); // undefined
console.log(userModule.validate); // undefined
// → True encapsulation via closure!
```

## §10. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 10.1 Pattern ①: 5 Whys — Selective Closure

```
5 WHYS: TẠI SAO CHỈ GIỮ BIẾN REFERENCED?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao V8 không giữ TẤT CẢ biến?
  └→ Vì biến không referenced = KHÔNG BAO GIỜ truy cập!
     Inner function là CÁCH DUY NHẤT truy cập backpack.
     Nếu inner không dùng → biến = dead code!

  WHY ②: V8 biết inner function dùng biến nào bằng cách nào?
  └→ Static analysis! V8 phân tích body code ở DEFINE TIME.
     Liệt kê tất cả identifiers → chỉ giữ danh sách đó.
     ⚠️ Ngoại trừ eval() → không thể static analyze!

  WHY ③: Tại sao tiết kiệm memory quan trọng?
  └→ Closure = data SỐNG MÃI! Nếu giữ tất cả →
     function gọi 1000 lần = 1000 closures = LEAK!
     Selective closure = giữ memory footprint nhỏ.

  WHY ④: Nếu inner function thay đổi (dynamic), thì sao?
  └→ Function body = CỐ ĐỊNH sau khi define!
     JS = interpreted nhưng function code = immutable.
     V8 chỉ cần đọc 1 lần = biết biến nào cần.

  WHY ⑤: eval() phá vỡ selective closure?
  └→ Đúng! eval("counter") = dynamic reference!
     V8 KHÔNG biết eval sẽ dùng biến gì →
     GIỮ TẤT CẢ! → eval = memory waste + security risk!
     ⭐ Thêm 1 lý do TRÁNH eval()!
```

### 10.2 Pattern ②: First Principles

```
FIRST PRINCIPLES — GARBAGE COLLECTION + CLOSURE:
═══════════════════════════════════════════════════════════════

  GC RULE: "Nếu KHÔNG CÓ REFERENCE → thu hồi!"

  Áp dụng vào closure:

  function outer() {
    let A = 1;  // ← inner dùng → có reference → SỐNG
    let B = 2;  // ← inner KHÔNG dùng → reference?
    function inner() { return A; }
    return inner;
  }
  const fn = outer();

  SAU KHI outer() kết thúc:
  ┌──────────────────────────────────────────────────────┐
  │ A = 1:                                               │
  │ → fn.[[Environment]] → reference đến A ✅            │
  │ → A CÓ reference → GC KHÔNG thu hồi!                │
  │                                                      │
  │ B = 2:                                               │
  │ → fn.[[Environment]] → KHÔNG reference đến B ❌      │
  │ → B KHÔNG CÓ reference nào!                          │
  │ → GC THU HỒI B! Xoá!                                │
  │                                                      │
  │ → Đây là "smart" operation:                          │
  │   V8 tạo [[Environment]] chỉ chứa {A: 1}            │
  │   KHÔNG phải {A: 1, B: 2}                            │
  └──────────────────────────────────────────────────────┘
```

### 10.3 Pattern ③: Trade-off Analysis

```
TRADE-OFFS: CLOSURE SIZE vs FUNCTIONALITY
═══════════════════════════════════════════════════════════════

  ┌──────────────────┬──────────────────┬──────────────────┐
  │ Approach         │ Memory           │ Flexibility      │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Keep ALL vars    │ ❌ Lãng phí     │ ✅ eval() works  │
  │ (no optimization)│ 100 biến x N fn │                  │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Keep REFERENCED  │ ✅ Tối thiểu    │ ⚠️ eval() breaks │
  │ (V8 default)     │ chỉ 2-3 biến   │ selective opt    │
  ├──────────────────┼──────────────────┼──────────────────┤
  │ Keep NOTHING     │ ✅✅ Không cost │ ❌ No closure!   │
  │ (no closure)     │                  │ Mất feature!     │
  └──────────────────┴──────────────────┴──────────────────┘

  V8's choice: Keep REFERENCED = sweet spot!
  → Tối thiểu memory ĐỦ ĐỂ closure hoạt động!
```

### 10.5 Pattern ⑤: Reverse Engineering

```javascript
// TỰ BUILD: Chứng minh selective closure thủ công

function testSelective() {
  let a = "a-value";
  let b = "b-value";
  let c = "c-value";

  // inner1 chỉ dùng a:
  function inner1() {
    return a;
  }

  // inner2 dùng a VÀ c:
  function inner2() {
    return a + c;
  }

  return { inner1, inner2 };
}

const { inner1, inner2 } = testSelective();

console.dir(inner1);
// [[Scopes]] → Closure(testSelective):
//   a: "a-value"       ← CHỈ a!
//   (b, c: KHÔNG CÓ!)

console.dir(inner2);
// [[Scopes]] → Closure(testSelective):
//   a: "a-value"       ← a VÀ c!
//   c: "c-value"
//   (b: KHÔNG CÓ!)

// → CÙNG outer function, KHÁC inner → KHÁC backpack!
// → V8 selective PER INNER FUNCTION!
```

### 10.6 Pattern ⑥: Lịch Sử

```
LỊCH SỬ: GARBAGE COLLECTION + CLOSURE OPTIMIZATION:
═══════════════════════════════════════════════════════════════

  2008: V8 Engine ra đời (Chrome)
  │     → Full closure (keep ALL variables from outer)
  │     → Performance concerns: closure = memory heavy?
  │
  ↓
  2010: V8 "Crankshaft" compiler
  │     → Bắt đầu optimize closures
  │     → Static analysis → selective closure!
  │     → Chỉ giữ referenced variables
  │
  ↓
  2015: V8 "TurboFan" compiler
  │     → Even better closure optimization
  │     → Inline closures khi có thể
  │     → Closure overhead ≈ 0 in many cases!
  │
  ↓
  2019: React Hooks phổ biến
  │     → Closures = EVERYWHERE
  │     → "Stale closure" = new bug category
  │     → Dan Abramov blog: "Understanding useEffect"
  │
  ↓
  2024: Modern V8
        → Closure optimization = mature
        → eval() = deopt trigger (giữ mọi thứ!)
        → with() = deopt trigger (giống eval!)
        → ⭐ Tránh eval/with = V8 optimize tốt nhất!
```

```
TÓM TẮT 6 PATTERNS:
═══════════════════════════════════════════════════════════════

  ① 5 WHYS         → Biến unreferenced = dead → GC xoá,
                      V8 static analysis tại define time,
                      eval() phá vỡ optimization!

  ② FIRST PRINCIPLES→ GC rule: no reference → collect,
                      [[Environment]] chỉ chứa referenced vars

  ③ TRADE-OFFS     → Keep all: waste memory vs eval works
                      Keep referenced: optimal, eval breaks
                      Keep nothing: no closure!

  ④ MENTAL MAPPING → React useRef persistent, DB pool private,
                      setTimeout callback giữ data

  ⑤ RE-IMPLEMENT   → console.dir chứng minh selective,
                      cùng outer + khác inner = khác backpack!

  ⑥ HISTORY        → V8 2008 (full) → Crankshaft (selective)
                      → TurboFan (inline) → mature 2024
```
