```
  │                                                      │
  │  ❌ SAI MENTAL MODEL:                                │
  │                                                      │

  │                                        = add1 + 🎒   │
  │                                        (ĐỘC LẬP!)   │
  │                                                      │
  │  → newFunc chỉ có: function code + backpack data    │
  │  → newFunc KHÔNG có liên kết ngược về outer         │
  │  → outer bị gọi lại → KHÔNG ảnh hưởng newFunc      │
  │  → outer bị xoá → newFunc VẪN hoạt động!            │
  │                                                      │
  └──────────────────────────────────────────────────────┘

  Will: "Go do the work of running outer. Whatever you
```

### 3.2 Chứng Minh: Xoá outer → newFunc Vẫn Hoạt Động!

```javascript
function outer() {
  let counter = 0;
  return function add1() {
    counter++;
    return counter;
  };
}

const newFunc = outer();

// XOÁ outer!
outer = null; // outer giờ = null, KHÔNG CÒN function!

// newFunc VẪN HOẠT ĐỘNG!
console.log(newFunc()); // 1
console.log(newFunc()); // 2
console.log(newFunc()); // 3

// → newFunc KHÔNG CẦN outer!
// → newFunc KHÔNG "referring back" to outer!
// → newFunc = FUNCTION CODE + BACKPACK (self-contained!)
// → outer = chỉ là "factory" — product sống ĐỘC LẬP!
```

## §4. Tenzin's Question: Khi Nào Backpack Bị Free?

### 4.1 Bài Viết: Backpack Sống Khi Nào?

> Tenzin: _"Does the memory allocated by the backpacks get freed when the program ends?"_
>
> Will: _"When the program ends, anything gets freed. But as long as the label anotherFunction and newFunc remain... If you were to reassign anotherFunction = null, at that point, the garbage collection would happen and these would be deleted."_

```
VÒNG ĐỜI CỦA BACKPACK:
═══════════════════════════════════════════════════════════════

  ① TẠO: Khi outer() chạy và trả ra inner function
     → Backpack được tạo + gắn vào returned function

  ② SỐNG: Khi CÒN REFERENCE đến function đó
     → const newFunc = outer(); → newFunc giữ ref → SỐNG
     → Dùng bao lâu → sống bấy lâu!

  ③ CHẾT: Khi KHÔNG CÒN REFERENCE nào đến function
     → newFunc = null; → mất reference → GC thu hồi!
     → Program kết thúc → tất cả freed!

  TIMELINE:
  ┌──────────────────────────────────────────────────────┐
  │ const newFunc = outer();  → 🎒 SỐNG!                │
  │ newFunc();                → 🎒 SỐNG, counter++      │
  │ newFunc();                → 🎒 SỐNG, counter++      │
  │ ...                                                  │
  │ // Nhiều dòng code khác, 🎒 VẪN SỐNG!               │
  │ ...                                                  │
  │ newFunc = null;           → ❌ Mất reference!       │
  │                           → GC: "Không ai dùng? XOÁ!"│
  │                           → 🎒 BỊ THU HỒI!          │
  └──────────────────────────────────────────────────────┘

  NHIỀU REFERENCES:
  const newFunc = outer();
  const alias = newFunc;     // 2 references!

  newFunc = null;            // 🎒 VẪN SỐNG! (alias còn!)
  alias = null;              // 🎒 BỊ THU HỒI! (0 refs!)
```

## §5. Memory Leak? — Không Phải "Inadvertent Memory Leak"

### 5.1 Bài Viết: True Memory Leak vs Closure "Leak"

Will đưa ra phân tích rất rõ ràng:

> _"A memory leak would be explicitly: you have data that CAN NO LONGER BE ACCESSED and yet is still taking up space. This WOULDN'T be that, because you definitionally CAN still access this data when you run anotherFunction — otherwise it won't be in here, because JavaScript would have explicitly not persisted it."_

```
TRUE MEMORY LEAK vs CLOSURE "LEAK":
═══════════════════════════════════════════════════════════════

  ❌ TRUE MEMORY LEAK (BUG THẬT SỰ):
  ┌──────────────────────────────────────────────────────┐
  │ Data CÒN trong memory                               │
  │ NHƯNG KHÔNG CÓ CÁCH NÀO truy cập được nữa!         │
  │ → "Data that can no longer be accessed               │
  │    and yet is still taking up space"                  │
  │                                                      │
  │ Ví dụ THẬT (DOM event listener leak):                │
  │ function addHandler() {                              │
  │   const hugeData = new Array(1e6);                   │
  │   button.addEventListener("click", () => {           │
  │     console.log(hugeData.length);                    │
  │   });                                                │
  │ }                                                    │
  │ // button bị remove khỏi DOM                         │
  │ // NHƯNG event listener VẪN reference hugeData!      │
  │ // hugeData VẪN trong memory = TRUE LEAK!            │
  └──────────────────────────────────────────────────────┘

  ✅ CLOSURE "LEAK" (KHÔNG PHẢI BUG - ĐÚNG THIẾT KẾ):
  ┌──────────────────────────────────────────────────────┐
  │ Data CÒN trong memory (backpack)                     │
  │ VÀ BẠN CÓ THỂ truy cập (bằng cách chạy function!)  │
  │ → "You definitionally CAN still access this data     │
  │    when you run anotherFunction"                     │
  │                                                      │
  │ const fn = outer(); // 🎒{ counter: 0 }              │
  │ fn(); // truy cập counter → ✅ CAN ACCESS!          │
  │ // → KHÔNG phải memory leak!                         │
  │ // → Đây là TÍNH NĂNG, không phải bug!               │
  └──────────────────────────────────────────────────────┘

  Will: "You can't have an INADVERTENT memory leak.
  You COULD have a sort of colloquial memory leak in
  the sense of you didn't realize you've got a bunch
  of data in there — but that's a problem you could
  have anyway."

  ⚠️ V8 GIỮ CHỈ DATA CẦN THIẾT:
  → "JavaScript does our optimization for us."
  → Selective closure: chỉ giữ referenced vars!
  → Nếu không referenced → already deleted!
  → "You saved the function at where you declare
     the data, so hopefully you know."
```

### 5.2 Khi Nào Closure THẬT SỰ Gây Memory Issue?

```
KHI NÀO CẦN CHÚ Ý:
═══════════════════════════════════════════════════════════════

  ⚠️ TÌNH HUỐNG 1: Closure trong event listener bị quên
  const listeners = [];
  for (let i = 0; i < 1000; i++) {
    const hugeBuffer = new ArrayBuffer(1e8); // 100MB!
    listeners.push(() => hugeBuffer.byteLength);
  }
  // 1000 closures × 100MB = 100GB backpack data! 💥
  // → Giải pháp: listeners = []; hoặc .removeEventListener()

  ⚠️ TÌNH HUỐNG 2: setInterval không clearInterval
  function startPolling() {
    const data = fetchHugeData(); // 50MB
    setInterval(() => {
      process(data); // data SỐNG MÃI trong backpack!
    }, 1000);
  }
  // → setInterval callback giữ reference → data SỐNG MÃI!
  // → Giải pháp: clearInterval khi không cần!

  ✅ TÌNH HUỐNG BÌNH THƯỜNG: Không vấn đề!
  function createCounter() {
    let count = 0; // 8 bytes — không đáng lo!
    return () => ++count;
  }
  // → 8 bytes trong backpack = nothing!
```

## §6. Chris's Insight: outer()() — IIFE & Module Pattern

### 6.1 Bài Viết: Gọi Ngay Kết Quả!

Chris nhận ra: _"Since outer returns a function, you could just call it again with more parentheses!"_

> Will: _"That's EXACTLY how the traditional module pattern worked — they allow you to immediately invoke and immediately create a memory space that is PERSISTENT and PRIVATE for all your code."_

```
CHRIS'S INSIGHT: outer()()
═══════════════════════════════════════════════════════════════

  Thay vì:
  const newFunc = outer(); // Bước 1: lấy function
  newFunc();               // Bước 2: gọi function

  Có thể:
  outer()();               // Gọi NGAY kết quả!
  // outer() → return ƒ → ƒ() → counter++ → 1

  GIẢI THÍCH:
  ┌──────────────────────────────────────────────────────┐
  │ outer()()                                            │
  │ ├── outer()  = gọi outer → return ƒ add1            │
  │ └── ()       = gọi ƒ add1 NGAY LẬP TỨC!            │
  │                                                      │
  │ Chained: outer()()() = ???                           │
  │ → outer() = ƒ add1                                  │
  │ → ƒ add1() = counter++ = 1 (return undefined)       │
  │ → undefined() = TypeError! ❌                        │
  │ → (vì add1 không return function!)                   │
  └──────────────────────────────────────────────────────┘

  NHƯNG nếu thay đổi code → IIFE Pattern:

  // IIFE = Immediately Invoked Function Expression
  const result = (function() {
    let counter = 0;
    return function add1() {
      counter++;
      return counter;
    };
  })();  // ← () NGAY SAU = invoke immediately!

  result(); // 1
  result(); // 2
  // → KHÔNG CẦN label cho outer function!
  // → Chạy 1 lần → lấy inner function + backpack!
```

### 6.2 Module Pattern — IIFE + Closure

```
MODULE PATTERN = IIFE + CLOSURE:
═══════════════════════════════════════════════════════════════

  Will: "They allow you to IMMEDIATELY INVOKE and
  immediately create a MEMORY SPACE that is persistent
  and private for all your code that runs within
  potentially just a large module function."

  const appModule = (function() {
    // ═══ PRIVATE STATE (backpack) ═══
    let users = [];
    let nextId = 1;
    const API_KEY = "secret-key-123";

    // ═══ PRIVATE FUNCTIONS ═══
    function validate(user) {
      return user.name && user.name.length > 0;
    }

    // ═══ PUBLIC API (returned object) ═══
    return {
      addUser(name) {
        const user = { id: nextId++, name };
        if (validate(user)) {
          users.push(user);
          return user;
        }
        throw new Error("Invalid user!");
      },
      getUsers() {
        return [...users]; // Return copy!
      },
      getUserCount() {
        return users.length;
      }
    };
  })();

  // SỬ DỤNG:
  appModule.addUser("Will");  // { id: 1, name: "Will" }
  appModule.addUser("Joe");   // { id: 2, name: "Joe" }
  appModule.getUserCount();   // 2

  // PRIVATE — KHÔNG truy cập được:
  appModule.users;     // undefined ❌
  appModule.nextId;    // undefined ❌
  appModule.API_KEY;   // undefined ❌
  appModule.validate;  // undefined ❌

  → "Initialization function that lays out all state"
  → "A wrapper function that you can ongoing use"
  → "That's essentially what the module pattern is doing"
```

## §7. Tổng Kết: 3 Loại Bộ Nhớ Trong Chương Trình JS

```
3 LOẠI BỘ NHỚ — TỔNG KẾT PHẦN CLOSURE:
═══════════════════════════════════════════════════════════════

  ① GLOBAL MEMORY:
  ┌──────────────────────────────────────────────────────┐
  │ Vòng đời: từ đầu đến cuối chương trình              │
  │ Ai access: MỌI CODE!                                │
  │ Ví dụ: const outer = ƒ, const newFunc = ƒ           │
  │ Đặc điểm: PUBLIC, SHARED, PERSISTENT                │
  │ Vấn đề: namespace pollution, no privacy!             │
  └──────────────────────────────────────────────────────┘

  ② LOCAL MEMORY (Execution Context):
  ┌──────────────────────────────────────────────────────┐
  │ Vòng đời: tạo khi function CHẠY, xoá khi KẾT THÚC │
  │ Ai access: code TRONG function đó                   │
  │ Ví dụ: let counter = 0 trong outer()                │
  │ Đặc điểm: PRIVATE, TEMPORARY, FRESH mỗi call       │
  │ Vấn đề: KHÔNG persistent! Quên hết khi EC xoá!     │
  └──────────────────────────────────────────────────────┘

  ③ CLOSURE MEMORY (Backpack / [[Scope]]):
  ┌──────────────────────────────────────────────────────┐
  │ Vòng đời: tạo khi function DEFINE, xoá khi GC      │
  │ Ai access: CHỈ function có backpack đó!             │
  │ Ví dụ: counter trong newFunc's backpack              │
  │ Đặc điểm: PRIVATE + PERSISTENT = BEST OF BOTH!     │
  │ Vấn đề: hidden complexity, hard to debug             │
  └──────────────────────────────────────────────────────┘

  BẢNG SO SÁNH:
  ┌──────────────┬──────────┬──────────┬──────────────────┐
  │              │ Global   │ Local    │ Closure (🎒)      │
  ├──────────────┼──────────┼──────────┼──────────────────┤
  │ Persistent?  │ ✅ Yes   │ ❌ No    │ ✅ Yes            │
  ├──────────────┼──────────┼──────────┼──────────────────┤
  │ Private?     │ ❌ No    │ ✅ Yes*  │ ✅ Yes            │
  ├──────────────┼──────────┼──────────┼──────────────────┤
  │ Between calls│ ✅ Yes   │ ❌ Fresh │ ✅ Yes            │
  ├──────────────┼──────────┼──────────┼──────────────────┤
  │ Ideal for    │ Constants│ Temp work│ STATE!           │
  └──────────────┴──────────┴──────────┴──────────────────┘
  * Private during execution, but temporary

  → CLOSURE = Private + Persistent = SWEET SPOT! 🎯
```

## §8. Tự Implement: GC, Null Assignment & Module Pattern

```javascript
// CHỨNG MINH 1: Xoá outer → newFunc vẫn hoạt động!

let outer = function () {
  let counter = 0;
  return function add1() {
    return ++counter;
  };
};

const newFunc = outer();
outer = null; // XOÁ outer hoàn toàn!

console.log(newFunc()); // 1 ← VẪN HOẠT ĐỘNG!
console.log(newFunc()); // 2 ← VẪN PERSISTENT!
console.log(newFunc()); // 3

// → Chứng minh: newFunc KHÔNG liên hệ với outer!
```

```javascript
// CHỨNG MINH 2: Set null → GC thu hồi backpack

let fn = (function () {
  const hugeData = new Array(1e6).fill("data"); // ~8MB
  return function () {
    return hugeData.length;
  };
})();

console.log(fn()); // 1000000 — backpack giữ 8MB
fn = null; // Mất reference → GC sẽ thu hồi 8MB!
// hugeData giờ KHÔNG CÒN ai reference → FREE!
```

```javascript
// CHỨNG MINH 3: WeakRef — cách track GC

let fn = (function () {
  const data = { value: 42 };
  return function () {
    return data.value;
  };
})();

// Trong thực tế, dùng FinalizationRegistry:
const registry = new FinalizationRegistry((msg) => {
  console.log(`${msg} has been garbage collected!`);
});

// Track đối tượng nào đó (không phải closure trực tiếp):
const obj = { fn };
registry.register(obj, "myObject");

fn = null;
// obj.fn = null;
// obj = null;
// → Khi GC chạy → "myObject has been garbage collected!"
```

```javascript
// CHỨNG MINH 4: Module pattern đầy đủ — real-world

const TaskManager = (function () {
  // === PRIVATE STATE (backpack) ===
  const tasks = [];
  let nextId = 1;
  let completedCount = 0;

  // === PRIVATE HELPERS ===
  function findTask(id) {
    return tasks.find((t) => t.id === id);
  }

  function formatTask(task) {
    return `[${task.done ? "✅" : "⬜"}] #${task.id}: ${task.title}`;
  }

  // === PUBLIC API ===
  return {
    add(title) {
      const task = {
        id: nextId++,
        title,
        done: false,
        createdAt: new Date(),
      };
      tasks.push(task);
      return formatTask(task);
    },

    complete(id) {
      const task = findTask(id);
      if (!task) throw new Error(`Task #${id} not found!`);
      if (task.done) return `Task #${id} already completed!`;
      task.done = true;
      completedCount++;
      return formatTask(task);
    },

    list() {
      return tasks.map(formatTask);
    },

    stats() {
      return {
        total: tasks.length,
        completed: completedCount,
        pending: tasks.length - completedCount,
      };
    },
  };
})(); // ← IIFE: chạy NGAY, tạo persistent private state!

// SỬ DỤNG:

TaskManager.add("Build a project"); // "[⬜] #2: Build a project"
TaskManager.complete(1); // "[✅] #1: Learn closures"
console.log(TaskManager.stats()); // { total: 2, completed: 1, pending: 1 }

// PRIVATE — KHÔNG truy cập được:

console.log(TaskManager.nextId); // undefined ❌
```

## §9. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 9.1 Pattern ①: 5 Whys — Zero Relationship

```
5 WHYS: TẠI SAO newFunc KHÔNG LIÊN HỆ VỚI outer?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao newFunc không "refer back" to outer?
  └→ Vì newFunc = returned VALUE, không phải function call!
     Giống: const x = add(1,2); → x = 3, không liên hệ add!

  WHY ②: Nhưng newFunc có data từ outer — không phải liên hệ?
  └→ Không! Data được COPY REFERENCE ra lúc return!
     Giống: nhận bằng tốt nghiệp → mang đi → trường đóng cửa!
     Bằng VẪN hợp lệ dù trường không còn!

  WHY ③: Vậy outer có biết newFunc đang dùng data không?
  └→ KHÔNG! outer = stateless function!
     outer KHÔNG giữ reference đến returned functions!
     outer KHÔNG biết ai đang dùng data!

  WHY ④: Nếu gọi outer() lần 3, có ảnh hưởng newFunc?
  └→ KHÔNG! outer() lần 3 = EC MỚI = counter MỚI!
     Mỗi outer() call = factory tạo product MỚI!
     Products ĐỘC LẬP!

  WHY ⑤: Vậy closure link ở đâu?
  └→ Link = GIỮA inner function VÀ enclosing DATA!
     [[Environment]] trỏ đến data (LexicalEnvironment)
     KHÔNG trỏ đến outer function!
     ⭐ Link = function ↔ data, KHÔNG PHẢI child ↔ parent!
```

```
FIRST PRINCIPLES — GC CỦA CLOSURE:

     → Bắt đầu từ "roots" (global, stack)
```

```javascript
// TỰ BUILD: Simulate GC behavior với closures

class GarbageCollector {
  constructor() {
    this.heap = new Map(); // address → { data, refCount }
  }

  allocate(data) {
    const addr = `0x${Math.random().toString(16).slice(2, 6)}`;
    this.heap.set(addr, { data, refCount: 1 });
    return addr;
  }

  addRef(addr) {
    const block = this.heap.get(addr);
    if (block) block.refCount++;
  }

  removeRef(addr) {
    const block = this.heap.get(addr);
    if (block) {
      block.refCount--;
      if (block.refCount === 0) {
        console.log(`GC: Freed ${addr} → ${JSON.stringify(block.data)}`);
        this.heap.delete(addr);
      }
    }
  }

  status() {
    console.log("=== HEAP STATUS ===");
    for (const [addr, block] of this.heap) {
      console.log(
        `  ${addr}: refs=${block.refCount}, data=${JSON.stringify(block.data)}`,
      );
    }
  }
}

const gc = new GarbageCollector();

// outer() call 1:
const bp1Addr = gc.allocate({ counter: 0 }); // newFunc's backpack
console.log("newFunc created");
gc.status();

// outer() call 2:
const bp2Addr = gc.allocate({ counter: 0 }); // anotherFunction's backpack
console.log("anotherFunction created");
gc.status();

// newFunc = null:
gc.removeRef(bp1Addr);
// "GC: Freed 0x... → {"counter":0}"
console.log("newFunc set to null");
gc.status();

// anotherFunction vẫn tồn tại:
// bp2 vẫn có refCount = 1 → SỐNG!
```
