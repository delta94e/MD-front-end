## §1. Code Phân Tích — outer + add1 + newFunc

```javascript
function outer() {
  let counter = 0;
  function add1() {
    counter++;
  }
  return add1;
}

const newFunc = outer();
newFunc(); // counter: 0 → 1
newFunc(); // counter: 1 → 2
```

## §4. The Backpack 🎒 — Dữ Liệu Đi Theo Function

### 4.1 Bài Viết: Nguồn Gốc Tên "Backpack"

Will chia sẻ một câu chuyện thú vị:

> _"Somebody in the audience once called this a backpack, and I will forever be grateful to them. So much so that I can't believe there's a very legitimate JavaScript book from O'Reilly that teaches closure, and on page 19, it refers to it as 'the backpack.' That made it into an official O'Reilly book!"_

```
BACKPACK — NHIỀU TÊN KHÁC NHAU:
═══════════════════════════════════════════════════════════════

  Tên chính thức (ECMAScript spec):
  → [[Environment]] — hidden property trên function object
  → Lexical Environment — environment nơi function được khai báo

  Tên Will dạy:
  → Backpack 🎒 — "túi ba lô" chứa data!

  Tên trong sách:
  → Closed-Over Variable Environment (C.O.V.E.)
  → Persistent Lexical Scope Referenced Data (P.L.S.R.D.)
  → Closure scope (Chrome DevTools)

  Tên đơn giản:
  → Closure — bản thân thuật ngữ chỉ TOÀN BỘ cơ chế

  Tại sao "Backpack" hay?
  ┌──────────────────────────────────────────────────────┐
  │ Giống sinh viên tốt nghiệp đại học:                  │
  │                                                      │
  │ → Trường (outer function) đã ĐÓNG CỬA (EC xoá)      │
  │ → Sinh viên (inner function) ra trường (return ra)   │
  │ → NHƯNG mang theo BA LÔ chứa kiến thức (data)!      │
  │ → Kiến thức (counter) KHÔNG BIẾN MẤT dù trường ĐÓNG│
  │ → Kiến thức CHỈ sinh viên đó có (private!)          │
  └──────────────────────────────────────────────────────┘
```

### 4.2 Backpack Chứa GÌ?

> Will: _"What comes into our backpack? Well, it is our counter. With the ACTUAL VALUE — not 'declare counter as zero' — but the actual stored counter, this zero in memory, is pulled out and attached with our function."_

```
BACKPACK CHỨA GÌ — CHÍNH XÁC:
═══════════════════════════════════════════════════════════════

  KHÔNG phải toàn bộ outer local memory!
  CHỈ các biến mà inner function THỰC SỰ REFERENCE!

  function outer() {
    let counter = 0;        ← REFERENCED bởi add1 ✅
    let unused = "hello";   ← KHÔNG referenced ❌
    let temp = [1,2,3];     ← KHÔNG referenced ❌
    function add1() {
      counter++;            ← DÙNG counter!
    }
    return add1;
  }

  BACKPACK 🎒 CỦA add1:
  ┌──────────────────────────────────────────────────────┐
  │ ✅ counter: 0     ← add1 DÙNG counter → kéo theo!  │
  │ ❌ unused         ← add1 KHÔNG dùng → BỎ ĐĂNG!     │
  │ ❌ temp           ← add1 KHÔNG dùng → BỎ ĐĂNG!     │
  │                                                      │
  │ → V8 Engine thông minh: chỉ giữ biến CẦN THIẾT!    │
  │ → Tiết kiệm memory!                                 │
  │ → Gọi là "selective closure"                         │
  └──────────────────────────────────────────────────────┘

  Will: "The ACTUAL VALUE, not 'declare counter as zero',
  but the actual stored counter — this zero IN MEMORY —
  is pulled out and attached."

  → Đây là LIVE reference, không phải copy!
  → Khi counter++ trong add1 → SỬA TRỰC TIẾP backpack!
  → Backpack = MUTABLE persistent store!
```

```
  │                                                      │

  │                                                      │
  │  ③ GLOBAL MEMORY:                                    │
  │     → Chỉ tìm ở đây nếu backpack CŨNG không có!     │

  SCOPE CHAIN THỰC TẾ (đầy đủ):
  Local → Closure(s) → Global → ReferenceError

  Có thể có NHIỀU closure scopes (nested closures):
  Local → Closure(inner) → Closure(middle) → Global

  Will: "Where does JavaScript's interpreter go look
  FIRST before it gets to global and gets sad?
  — In the BACKPACK."
```

## §8. Private Data & Lexical Bond

### 8.1 Bài Viết: Backpack = Private Data

> Will: _"Only available inside the running of newFunc. You could call it PRIVATE DATA."_

```
PRIVATE DATA — CHỈ FUNCTION ĐÓ ACCESS:
═══════════════════════════════════════════════════════════════

  const newFunc = outer();

  ✅ newFunc() → TRUY CẬP counter qua closure!
  ❌ console.log(counter) → ReferenceError! (global không có)
  ❌ newFunc.counter → undefined! (không phải property!)
  ❌ Object.keys(newFunc) → [] (không có gì!)

  → counter CHỈ tồn tại TRONG backpack 🎒 của newFunc!
  → KHÔNG AI KHÁC truy cập được!
  → Đây = ENCAPSULATION = PRIVATE STATE!
  → Giống private field trong class, NHƯNG không cần class!
```

### 8.2 Matt's Question: "Có Thể Nhìn Code Để Biết?"

Matt hỏi: _"If you're looking at the code, could you use the parent scope brackets as being part of that backpack?"_

Will trả lời:

> _"The term used for this has the word LEXICAL in it, and lexical comes from the code you actually wrote — not what happens when you run it, but where you SAVED it, where you positioned it on the page."_

```
LEXICAL BOND — NHÌN CODE = BIẾT CLOSURE:
═══════════════════════════════════════════════════════════════

  function outer() {           ← parent scope { }
    let counter = 0;           ← biến TRONG parent scope
    function add1() {          ← add1 VIẾT TRONG parent scope
      counter++;               ← REFERENCE biến của parent!
    }                          → Nhìn CODE = biết add1 có closure!
    return add1;
  }

  "Lexical" = "liên quan đến TEXT, code bạn VIẾT"
  → Nhìn vị trí add1 TRONG SOURCE CODE
  → add1 nằm TRONG outer → thấy biến CỦA outer
  → ĐÓ chính là lexical scope!
  → KHÔNG cần chạy code để biết — chỉ cần NHÌN!

  Will: "It is tied to where I DEFINED the function.
  But be careful — it's not just the curly braces. It's
  an EXECUTION of the code where we actually save LIVE
  counter = 0 in memory. At that moment we get a LIVE
  BOND to that surrounding data."

  ⚠️ QUAN TRỌNG:
  → "Live bond" = REFERENCE, không phải copy!
  → counter trong backpack = CÙNG counter trong outer!
  → Khi bạn counter++ → giá trị THAY ĐỔI trong backpack!
  → Backpack = MUTABLE!
```

## §9. Tự Implement: Closure Internals

```javascript
// TỰ BUILD: Chứng minh CLOSURE hoạt động

function outer() {
  let counter = 0;
  function add1() {
    counter++;
    console.log("counter =", counter);
  }
  return add1;
}

const newFunc = outer();

// Inspect closure bằng console.dir:
console.dir(newFunc);
// ƒ add1()
//   name: "add1"
//   length: 0
//   [[Scopes]]:                   ← ĐÂY LÀ BACKPACK!
//     0: Closure (outer)          ← Closure scope!
//       counter: 0                ← DATA TRONG BACKPACK!
//     1: Script {}
//     2: Global {window, ...}

// GỌI:
newFunc(); // "counter = 1"
newFunc(); // "counter = 2"
newFunc(); // "counter = 3"

// Inspect LẠI:
console.dir(newFunc);
//   [[Scopes]]:
//     0: Closure (outer)
//       counter: 3                ← ĐÃ CẬP NHẬT!
```

```javascript
// TỰ BUILD: 2 Closures ĐỘC LẬP

function outer() {
  let counter = 0;
  function add1() {
    counter++;
    return counter;
  }
```

## §10. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 10.1 Pattern ①: 5 Whys — Backpack

```
5 WHYS: TẠI SAO DATA SỐNG SÓT SAU KHI EC BỊ XOÁ?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao counter KHÔNG bị xoá khi outer kết thúc?
  └→ Vì add1 GIỮ REFERENCE đến counter!
     Garbage Collector rule: "nếu có reference → KHÔNG xoá!"

  WHY ②: Tại sao add1 giữ reference?
  └→ Vì khi add1 được DEFINE bên trong outer,
     JS tự động tạo hidden property [[Environment]]
     trỏ đến outer's lexical environment!

  WHY ③: Tại sao [[Environment]] tồn tại?
  └→ Vì ECMAScript spec yêu cầu: mỗi function object
     PHẢI lưu reference đến enclosing environment
     tại thời điểm CREATION (không phải call)!
     → Đây = LEXICAL SCOPE implementation!

  WHY ④: Tại sao return function = kéo data theo?
  └→ Vì return function = return OBJECT (first-class!)
     Object có [[Environment]] → reference đến outer data
     → outer data CÓ REFERENCE → GC KHÔNG xoá!
     → Data "đi theo" function! = BACKPACK!

  WHY ⑤: Tại sao gọi là "closure"?
  └→ Vì function "ĐÓNG LẠI" (close over) biến từ
     enclosing scope! Biến bị "nhốt" trong function!
     Peter Landin (1964): "The function together with
     its environment is called a CLOSURE."
     ⭐ Closure = function + environment = code + backpack!
```

### 10.5 Pattern ⑤: Reverse Engineering

```javascript
// TỰ BUILD: Giả lập đầy đủ cơ chế closure của V8

class LexicalEnvironment {
  constructor(outer = null) {
    this.bindings = {};
    this.outer = outer; // ← scope chain link!
  }

  define(name, value) {
    this.bindings[name] = value;
  }

  lookup(name) {
    // ① Tìm trong bindings hiện tại:
    if (name in this.bindings) {
      return { found: true, value: this.bindings[name], scope: this };
    }
    // ② Đi đến outer scope (LEXICAL, không phải call stack!):
    if (this.outer) {
      return this.outer.lookup(name);
    }
    // ③ Không có → Error!
    return { found: false, error: `ReferenceError: ${name} is not defined` };
  }

  update(name, value) {
    if (name in this.bindings) {
      this.bindings[name] = value;
      return true;
    }
    if (this.outer) return this.outer.update(name, value);
    return false;
  }
}

// Simulate: outer() → return add1 → newFunc()
// Bước 1: Global environment
const globalEnv = new LexicalEnvironment(null);
globalEnv.define("outer", "ƒ");

// Bước 2: outer() → tạo EC → local env
const outerEnv = new LexicalEnvironment(globalEnv);
outerEnv.define("counter", 0);
outerEnv.define("add1", "ƒ");

// Bước 3: add1 được DEFINE → [[Environment]] = outerEnv
// Khi return add1 → newFunc = ƒ + backpack(outerEnv)

// Bước 4: newFunc() → tạo EC mới → local env mới
const newFuncEnv = new LexicalEnvironment(outerEnv); // ← BACKPACK!
// outer = outerEnv (NƠI KHAI BÁO), KHÔNG PHẢI globalEnv!

// counter++ → tìm "counter":
console.log(newFuncEnv.lookup("counter"));
// { found: true, value: 0, scope: outerEnv } ← TÌM THẤY Ở BACKPACK!

newFuncEnv.update("counter", 1); // counter++
console.log(outerEnv.lookup("counter"));
// { found: true, value: 1 } ← BACKPACK CẬP NHẬT!
```
