## §1. Code Phân Tích — newFunc + anotherFunction

```javascript
function outer() {
  let counter = 0;
  function add1() {
    counter++;
  }
  return add1;
}

const newFunc = outer(); // Gọi outer LẦN 1
newFunc(); // counter: 0 → 1
newFunc(); // counter: 1 → 2

const anotherFunction = outer(); // Gọi outer LẦN 2!
anotherFunction(); // counter: ???
anotherFunction(); // counter: ???
```

Will nhấn mạnh sự chính xác ngôn ngữ:

> _"anotherFunction is NOT the call to outer. It's the EVALUATED RESULT, the RETURN VALUE of the second call to outer. You can see why that precision is so important when dealing with returning functions."_

## §3. Gọi outer() Lần 2 → anotherFunction → Backpack MỚI!

### 3.1 Bài Viết: "Brand New Execution Context!"

```
GỌI outer() LẦN 2 → BRAND NEW EVERYTHING!
═══════════════════════════════════════════════════════════════

  const anotherFunction = outer();

  Will: "We are going to call outer AGAIN. Brand new
  call to outer."

  ┌─────────────────────────────────────────────────────┐
  │                                                     │
  │ LẦN 1 (đã xong):                                   │
  │ outer() → EC #1 → counter₁: 0 → add1₁             │
  │ → return add1₁ → newFunc + 🎒₁{counter₁: 0}       │
  │ → EC #1 XOÁ   (nhưng 🎒₁ vẫn SỐNG!)              │
  │                                                     │
  │ LẦN 2 (bây giờ):                                   │
  │ outer() → EC #2 → counter₂: 0 → add1₂             │
  │ → return add1₂ → anotherFunction + 🎒₂{counter₂: 0}│
  │ → EC #2 XOÁ   (nhưng 🎒₂ vẫn SỐNG!)              │
  │                                                     │
  │ EC #1 ≠ EC #2!                                      │
  │ counter₁ ≠ counter₂!                                │
  │ 🎒₁ ≠ 🎒₂!                                         │
  │ → HOÀN TOÀN ĐỘC LẬP!                               │
  │                                                     │
  └─────────────────────────────────────────────────────┘
```

## §6. What-If: counter Ở Local → Luôn = 1!

> Will: _"What if I had counter = 0 defined in my add1 function? What happens?"_

```
WHAT-IF ①: COUNTER DEFINED TRONG LOCAL (add1):
═══════════════════════════════════════════════════════════════

  function outer() {
    let counter = 0;          // ← VẪN Ở ĐÂY (backpack)
    function add1() {
      let counter = 0;        // ← THÊM VÀO ĐÂY (local!)
      counter++;
      console.log(counter);
    }
    return add1;
  }

  const newFunc = outer();
  newFunc();           // → ???
  newFunc();           // → ???
  anotherFunction();   // → ???
  anotherFunction();   // → ???

  TRACE:
  ┌──────────────────────────────────────────────────────┐
  │ newFunc() lần 1:                                     │
  │   let counter = 0;   ← DEFINE trong LOCAL!          │
  │   counter++;                                         │
  │   → Tìm "counter" ở LOCAL → TÌM THẤY! (0)          │
  │   → counter++ → 1                                   │
  │   → console.log(1)                                   │
  │   → EC xoá → local counter BIẾN MẤT!                │
  │                                                      │
  │ newFunc() lần 2:                                     │
  │   let counter = 0;   ← LẠI DEFINE MỚI!             │
  │   counter++;                                         │
  │   → Tìm LOCAL → TÌM THẤY! (0 lại từ đầu!)          │
  │   → counter++ → 1                                   │
  │   → console.log(1)                                   │
  │   → EC xoá → BIẾN MẤT lần nữa!                     │
  └──────────────────────────────────────────────────────┘

  OUTPUT: 1, 1, 1, 1  ← LUÔN LÀ 1!

  Ryan: "You get 1 every time."
  Will: "Exactly! Because we grab counter in LOCAL,
  increment to 1. That execution context runs, gone.
  Do it again — 1."

  TẠI SAO?
  → "JavaScript would FIRST grab in local"
  → Lookup order: Local TRƯỚC Backpack!
  → Nếu local có counter → KHÔNG BAO GIỜ đến backpack!
  → counter trong backpack = DEAD CODE (không bao giờ dùng!)
```

## §8. Lookup Order Tổng Kết: Local → Backpack → Global

```
SO SÁNH 3 SCENARIOS — LOOKUP ORDER:
═══════════════════════════════════════════════════════════════

  SCENARIO A: counter TRONG CLOSURE (backpack):
  ┌──────────────────────────────────────────────────────┐
  │ Lookup: Local ❌ → Backpack ✅ (DỪNG!)              │
  │ Output: 1, 2 | 1, 2  ← ĐỘC LẬP! Private!          │
  │                                                      │
  │ → MỖI function có counter RIÊNG!                     │
  │ → Persistent! Private! Isolated!                     │
  │ → ĐÂY LÀ CLOSURE!                                   │
  └──────────────────────────────────────────────────────┘

  SCENARIO B: counter TRONG LOCAL:
  ┌──────────────────────────────────────────────────────┐
  │ Lookup: Local ✅ (DỪNG! Không đến backpack!)         │
  │ Output: 1, 1, 1, 1  ← "Reset" mỗi call!            │
  │                                                      │
  │ → counter KHÔNG persistent!                          │
  │ → Tạo mới + xoá MỖI call!                           │
  │ → Backpack counter = unreachable!                    │
  └──────────────────────────────────────────────────────┘

  SCENARIO C: counter TRONG GLOBAL:
  ┌──────────────────────────────────────────────────────┐
  │ Lookup: Local ❌ → Backpack ❌ → Global ✅           │
  │ Output: 1, 2, 3, 4  ← SHARED! Không private!        │
  │                                                      │
  │ → TẤT CẢ functions chia sẻ CÙNG counter!            │
  │ → Ai cũng sửa được! Không encapsulation!            │
  │ → BUG TIỀM ẨN: function khác sửa nhầm!             │
  └──────────────────────────────────────────────────────┘

  → CLOSURE (Scenario A) = SWEET SPOT!
  → Persistent (khác local) + Private (khác global)!
```

## §9. Tự Implement: Independent Closures & Scope Scenarios

```javascript
// CHỨNG MINH 1: Independent closures — 2 backpacks

function createCounter(name) {
  let count = 0;
  return function () {
    count++;
    console.log(`${name}: ${count}`);
    return count;
  };
}

const counterA = createCounter("A");
const counterB = createCounter("B");

counterA(); // "A: 1"
counterA(); // "A: 2"
counterA(); // "A: 3"

counterB(); // "B: 1" ← BẮT ĐẦU TỪ 0! Không phải 4!
counterB(); // "B: 2"

counterA(); // "A: 4" ← counterA vẫn nhớ! Không bị reset!

// → 2 closures = 2 backpacks = 2 cuộc sống ĐỘC LẬP!
```

```javascript
// CHỨNG MINH 2: 3 scenarios so sánh — cùng 1 lúc

// SCENARIO A: counter trong closure (backpack)
function makeClosureCounter() {
  let counter = 0;
  return function () {
    return ++counter;
  };
}

// SCENARIO B: counter trong local
function makeLocalCounter() {
  return function () {
    let counter = 0; // Reset mỗi call!
    return ++counter;
  };
}

// SCENARIO C: counter trong global
let globalCounter = 0;
function makeGlobalCounter() {
  return function () {
    return ++globalCounter;
  };
}

// Tạo 2 instances mỗi loại:
const closureA = makeClosureCounter();
const closureB = makeClosureCounter();
const localA = makeLocalCounter();
const localB = makeLocalCounter();
const globalA = makeGlobalCounter();
const globalB = makeGlobalCounter();

// Gọi 4 lần:
console.log("=== Closure (independent) ===");
console.log(closureA(), closureA()); // 1, 2
console.log(closureB(), closureB()); // 1, 2 ← ĐỘC LẬP!

console.log("=== Local (always 1) ===");
console.log(localA(), localA()); // 1, 1
console.log(localB(), localB()); // 1, 1 ← LUÔN = 1!

console.log("=== Global (shared) ===");
console.log(globalA(), globalA()); // 1, 2
console.log(globalB(), globalB()); // 3, 4 ← TIẾP TỤC!
```

```javascript
// CHỨNG MINH 3: Local variable SHADOWS closure variable

function outer() {
  let x = "closure";
  return function inner() {
    let x = "local"; // SHADOWS closure x!
    return x; // → "local", KHÔNG phải "closure"!
  };
}

const fn = outer();
console.log(fn()); // "local"

// → Local "x" shadows (che khuất) closure "x"
// → JavaScript DỪNG tìm khi thấy ở local!
// → Closure "x" = unreachable = wasted memory!
// → V8 VẪN giữ closure "x" (vì static analysis phức tạp)
// → Nhưng trong practice: ĐỪNG shadow closure vars!
```

```javascript
// CHỨNG MINH 4: Sử dụng thực tế — nhiều independent instances

function createPlayer(name) {
  let score = 0;
  let level = 1;

  return {
    addPoints(points) {
      score += points;
      if (score >= level * 100) {
        level++;
        console.log(`${name} leveled up to ${level}!`);
      }
    },
    getStats() {
      return { name, score, level };
    },
  };
}

const player1 = createPlayer("Will");
const player2 = createPlayer("Joe");
const player3 = createPlayer("Matt");

player1.addPoints(50);
player1.addPoints(60); // "Will leveled up to 2!"
player2.addPoints(30);
player3.addPoints(200); // "Matt leveled up to 2!" rồi tiếp...

console.log(player1.getStats()); // { name: "Will", score: 110, level: 2 }
console.log(player2.getStats()); // { name: "Joe", score: 30, level: 1 }
console.log(player3.getStats()); // { name: "Matt", score: 200, level: 3 }

// → 3 players = 3 closures = 3 backpacks ĐỘC LẬP!
// → Giống OOP instances nhưng KHÔNG CẦN class!
```

## §10. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 10.1 Pattern ①: 5 Whys — Independent Closures

```
5 WHYS: TẠI SAO MỖI outer() TẠO BACKPACK MỚI?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao anotherFunction có backpack riêng?
  └→ Vì gọi outer() LẦN 2 = tạo EC MỚI HOÀN TOÀN!
     counter MỚI = 0 (không liên quan counter cũ!)

  WHY ②: Tại sao EC mới = counter mới?
  └→ Vì mỗi EC có local memory RIÊNG!
     let counter = 0 chạy LẠI → biến MỚI trong RAM!
     Giống: mỗi lần gọi = 1 "chương trình mini" riêng!

  WHY ③: Tại sao backpack cũ không bị ảnh hưởng?
  └→ Vì backpack₁ (newFunc) và backpack₂ (anotherFunction)
     trỏ đến 2 vùng nhớ KHÁC NHAU trong RAM!
     Sửa 1 → KHÔNG ảnh hưởng cái kia!

  WHY ④: Vậy giống class instances?
  └→ Đúng! Mỗi outer() call = giống new MyClass()!
     Mỗi instance có state RIÊNG (backpack)!
     Closure = OOP-like encapsulation KHÔNG CẦN class!

  WHY ⑤: Khi nào cần SHARED closure?
  └→ Khi 2 functions cần chia sẻ CÙng data → return
     OBJECT chứa nhiều methods từ CÙNG 1 outer() call!
     = Module Pattern! (sẽ thấy sau!)
```

### 10.2 Pattern ②: First Principles

```
FIRST PRINCIPLES — TẠI SAO ĐỘC LẬP?
═══════════════════════════════════════════════════════════════

  JS Engine (V8) xử lý MỖI outer() call:

  Call 1: outer()
  ① Allocate VÙNG NHỚ MỚI trong heap
  ② Tạo LexicalEnvironment mới { counter: 0 }
  ③ Define add1 → [[Environment]] = LexicalEnv₁
  ④ Return add1 → newFunc giữ ref đến LexicalEnv₁
  ⑤ Pop EC → LexicalEnv₁ SỐNG NHỜ newFunc's ref

  Call 2: outer()
  ① Allocate VÙNG NHỚ KHÁC trong heap      ← KHÁC!
  ② Tạo LexicalEnvironment mới { counter: 0 } ← MỚI!
  ③ Define add1 → [[Environment]] = LexicalEnv₂  ← KHÁC!
  ④ Return add1 → anotherFunction ref → LexicalEnv₂
  ⑤ Pop EC → LexicalEnv₂ SỐNG NHỜ anotherFunction

  HEAP:
  ┌─────────────────────────────────────┐
  │ Address 0x1000: { counter: 2 } ← 🎒₁ (newFunc)      │
  │ Address 0x2000: { counter: 2 } ← 🎒₂ (anotherFunc)  │
  │                                                       │
  │ → 2 VÙNG NHỚ KHÁC NHAU!                              │
  │ → Sửa 0x1000 ≠ sửa 0x2000!                           │
  └─────────────────────────────────────┘
```

### 10.3 Pattern ③: Trade-off Analysis

```
TRADE-OFFS: INDEPENDENT vs SHARED CLOSURES
```

### 10.5 Pattern ⑤: Reverse Engineering

```javascript
// TỰ BUILD: Giả lập V8 tạo multiple closures

class HeapMemory {
  constructor() {
    this.blocks = {};
    this.nextAddress = 0x1000;
  }

  allocate(data) {
    const addr = this.nextAddress;
    this.blocks[addr] = { ...data };
    this.nextAddress += 0x1000;
    return addr;
  }

  read(addr, key) {
    return this.blocks[addr][key];
  }
  write(addr, key, value) {
    this.blocks[addr][key] = value;
  }
}

const heap = new HeapMemory();

// Gọi outer() lần 1:
const backpack1Addr = heap.allocate({ counter: 0 });
// newFunc = { code: "counter++", backpackAddr: 0x1000 }

// Gọi outer() lần 2:
const backpack2Addr = heap.allocate({ counter: 0 });
// anotherFunction = { code: "counter++", backpackAddr: 0x2000 }

// newFunc(): counter++ trên backpack 1
heap.write(backpack1Addr, "counter", heap.read(backpack1Addr, "counter") + 1);
console.log("newFunc #1:", heap.read(backpack1Addr, "counter")); // 1

heap.write(backpack1Addr, "counter", heap.read(backpack1Addr, "counter") + 1);
console.log("newFunc #2:", heap.read(backpack1Addr, "counter")); // 2

// anotherFunction(): counter++ trên backpack 2 (ĐỘC LẬP!)
heap.write(backpack2Addr, "counter", heap.read(backpack2Addr, "counter") + 1);
console.log("another #1:", heap.read(backpack2Addr, "counter")); // 1 ← TỪ 0!

heap.write(backpack2Addr, "counter", heap.read(backpack2Addr, "counter") + 1);
console.log("another #2:", heap.read(backpack2Addr, "counter")); // 2

// → 2 địa chỉ KHÁC NHAU trong heap = 2 backpacks ĐỘC LẬP!
```

### 10.6 Pattern ⑥: Lịch Sử

```
LỊCH SỬ: MULTIPLE CLOSURES → OOP ALTERNATIVE:
═══════════════════════════════════════════════════════════════

  1990s: OOP dominates — class = cách duy nhất tạo instances
  │     → new Player("Will") → instance có state riêng
  │     → Cần class/constructor/this/prototype = phức tạp!
  │

  │     → Closures = thay thế classes cho encapsulation!
  │     → Mỗi outer() call = 1 "instance" với private state!
  │     → "Closures are the better pattern for JavaScript"
  │
  ↓
  2008: Module Pattern phổ biến
  │     → IIFE + closure = module with private state
  │     → jQuery, Backbone đều dùng pattern này!
  │
  ↓
  2015: ES6 Classes = syntactic sugar
  │     → Nhưng bên dưới VẪN dùng prototypes, closures!
  │     → Factory functions (closure) vs Classes = cuộc tranh luận!
  │
  ↓
  2019: React Hooks = CLOSURES THẮNG!
  │     → useState, useEffect = tất cả đều closure-based
  │     → Mỗi component render = closure instance!
  │     → "Classes are dead" (trong React ecosystem)
  │
  ↓
  2024: Closure-based patterns = mainstream
        → Factory functions, Hooks, Module pattern
        → Classes vẫn tồn tại nhưng closures = preferred
        → ⭐ Multiple closures = multiple instances = OOP!
```
