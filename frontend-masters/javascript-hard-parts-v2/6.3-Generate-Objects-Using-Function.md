## §2. Solution 1: Wrap In A Function — userCreator

```javascript
function userCreator(name, score) {
  const newUser = {};
  newUser.name = name;
  newUser.score = score;
  newUser.increment = function () {
    newUser.score++;
  };
  return newUser;
}

const user1 = userCreator("Ari", 3);
const user2 = userCreator("J", 5);

user1.increment();
```

```
FACTORY FUNCTION PATTERN:
═══════════════════════════════════════════════════════════════

  userCreator("Ari", 3)
       │
       ▼
  ┌──────────────────────────────────────────────────┐
  │ FACTORY:                                         │
  │                                                  │
  │ ① Create empty object: {}                       │
  │ ② Add name property: "Ari"                      │
  │ ③ Add score property: 3                         │
  │ ④ Add increment method: ƒ                       │
  │ ⑤ Return the complete object!                   │
  │                                                  │
  └──────────────────────────────────────────────────┘
       │
       ▼
  user1 = { name: "Ari", score: 3, increment: ƒ }

  → Pass in details → get back complete object!
  → No more manual creation!
  → DRY ✅!
```

// CHỨNG MINH 2: Methods are SEPARATE (closure proof)

function userCreator(name, score) {
  const newUser = {};
  newUser.name = name;
  newUser.score = score;
  newUser.increment = function () {
    newUser.score++; // references newUser via CLOSURE!

```javascript
// CHỨNG MINH 4: Closure vs this — understanding the reference

// Will's version (closure → newUser):
function creatorWithClosure(name, score) {
  const newUser = {};
  newUser.name = name;
  newUser.score = score;
  newUser.increment = function () {
    newUser.score++; // → uses CLOSURE to find newUser!
  };
  return newUser;
}

// Alternative with `this`:
function creatorWithThis(name, score) {
  const newUser = {};
  newUser.name = name;
  newUser.score = score;
  newUser.increment = function () {
    this.score++; // → uses `this` (determined at call time!)
  };
  return newUser;
}

const a = creatorWithClosure("A", 1);
const b = creatorWithThis("B", 1);

a.increment(); // works! (closure finds newUser)
b.increment(); // works! (this = b when called as b.increment())

console.log(a.score); // 2 ✅
console.log(b.score); // 2 ✅

// Both work, but:
// Closure version: locked to specific object (via closure)
// `this` version: would allow sharing (if we could share the function)
// → `this` version is needed for prototype chain later!
```

## §8. 🔬 Deep Analysis Patterns — 6 Tư Duy Phân Tích Sâu

### 8.1 Pattern ①: 5 Whys — Factory Function

```
5 WHYS: TẠI SAO DÙNG FACTORY FUNCTION?
═══════════════════════════════════════════════════════════════

  WHY ①: Tại sao wrap object creation trong function?
  └→ Vì DRY! Don't repeat creating objects manually!
     1 function → millions of objects!

  WHY ②: Tại sao return object thay vì modify global?
  └→ Vì ENCAPSULATION! Object sống riêng!
     Mỗi call tạo INDEPENDENT object!
     No shared state between users!

  WHY ③: Tại sao increment dùng newUser (closure)?
  └→ Vì khi define, newUser là only reference!
     `this` chưa được introduce ở đây!
     Closure = safe way to reference the object!

  WHY ④: Tại sao vẫn chưa đủ tốt?
  └→ Vì methods COPIED mỗi lần call!
     5 methods × 1M users = 5M function objects!
     Same code, different memory! WASTEFUL!

  WHY ⑤: Tại sao cần giải pháp khác?
  └→ Vì Goal ③ (performant) chưa đạt!
     Need: SHARE methods without copying!
     = Prototype chain (next lesson!)
```

### 8.2 Pattern ②: First Principles

```
FIRST PRINCIPLES — FACTORY FUNCTION:
═══════════════════════════════════════════════════════════════

  Factory Pattern (Gang of Four):
  → Create objects WITHOUT specifying exact class!
  → Delegate creation to a function!
  → Function = the "factory" that builds objects!

  In JavaScript:
  → function = factory
  → {} = raw material
  → dot notation = assembly
  → return = delivery!

  EACH CALL = BRAND NEW:
  → Brand new execution context!
  → Brand new local memory!
  → Brand new empty object!
  → Brand new method copies! ← PROBLEM!

  CLOSURE in Factory Functions:
  → Methods defined inside factory = closure!
  → Each method's backpack has reference to its newUser!
  → SEPARATE backpacks for SEPARATE users!
  → Correct behavior but wasteful memory!
```
